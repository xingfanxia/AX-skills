# 11 — 二游生产与长线运营（把剧情做成能持续供货的内容机器）

> Purpose: 这是**生产与长线运营机制(production & live-service machinery)**层。前面的 reference 教"怎么把一段故事写好"（ref 01 结构 / ref 02 角色 / ref 03 世界观 / ref 07 craft）；这一份教"当创作者从『写好一段故事』升级到『把它做成一个能长期运营、按版本供货、还要顺便卖角色的项目』时，需要的那套**项目级机器**"。主要用在**二游规划阶段**与 SKILL.md 的 **S4+（主线框架之后）**。与 ref 05（CN/米哈游系结构）、ref 06（日韩传统 + 选边）、ref 02（角色）互补——那三份讲"内容本身怎么写"，本份讲"内容怎么被规划、记忆、排期、交付与运营"。
>
> 用法：本份**不是给新手一次填满的清单**，而是给 AI 教练随创作进度逐步维护的工具箱。每节结构固定：**原理/为什么 → 模板或表 → 可复用要点**。所有命名案例、日期、数字、URL 与 ⚠️ 不确定性标注均保留，不臆造。

## 1. Canon 管理 + 项目状态(Project State) —— 多 session / 长线创作的记忆系统

### 原理 / 为什么

二游创作天然是**跨多次会话(multi-session)、跨数月甚至数年**的。一个真实的副作用是：**设定会漂移、自相矛盾**——这次会话定的"主角失忆"，下次会话不知不觉写成了"主角只是装失忆"；某个配角的姓在 S5 是一个写法，到 S6 又变了。线性媒体（小说、电影）靠一个作者的连续记忆兜底；二游靠的是一份**活文档(living doc)作为单一事实来源(single source of truth)**。

> 这正是 ref 05 §1.3 与 ref 06 §1.1 反复强调的"防崩骨架"在**单人 / 小团队 + AI 协作**场景下的落地：米哈游靠"稳定小团队脑内共享 + 总编辑头尾统一改写"，奈须靠"伏笔账本 + 亲自改每章头尾"。你（AI）一个人协作时，没有团队的脑内共享，**那份活文档就是你的脑内共享**。

**核心机制是给每条设定打"状态标签"。** 不要把所有设定平铺成一坨"设定集"——那样无法判断哪条已经板上钉钉、哪条还能改。用 **Canon 四态**：

| 状态 | 含义 | AI 的行为 |
|---|---|---|
| **Canon（已定稿）** | 创作者已确认，后续所有内容必须遵守 | 视为不可违背；新内容与它冲突时**必须报警** |
| **Pending（待确认）** | 正在探索、尚未拍板，可继续发散 | 可以继续在它上面构思，但标注"未定，可能推翻" |
| **Rejected（已否决）** | 创作者明确不要的方向 | **不要反复提出**；除非创作者主动重启该方向 |
| **Idea Bank（备选灵感）** | 暂不进主线的好点子，存着备用 | 不主动塞进当前剧情；合适时机提醒"我们存过一个…" |

**冲突处理铁律**：当一个新想法与某条 **Canon** 冲突，AI **不能默默采纳新想法、也不能默默丢弃它**，而要**显式指出冲突**并给三种处理选项让创作者拍板：
1. **改新想法**——让它兼容已定的 Canon；
2. **改旧设定**——把那条 Canon 降级回 Pending 再调整（并提示这会波及哪些已写内容）；
3. **把冲突变成剧情冲突**——很多"设定打架"其实是好故事的种子（"两份记录互相矛盾"本身可以是一条悬念）。

### 模板 —— Project State（精简 YAML）

这份 YAML 是**项目的中枢档案**，由 AI 维护、随创作累积，不要求创作者一次填满。空字段就是"还没到那一步"。

```yaml
meta:
  title:            # 项目代号 / 暂定名
  creator_role:     # 创作者自我定位（如：独立开发 / 同人 / 商业团队编剧）
  project_stage:    # S0 校准 / S1 种子 / … / S6 写场景 / live_ops 运营
  business_model:   # 买断 / 抽卡 / 订阅 / 免费+广告 / 未定

language:
  communication:    # 与创作者交流语言（默认 中文）
  deliverable:      # 产出文档语言
  terminology_policy: # 术语策略（默认：首次中文+括注英文，之后只用中文）
  glossary:         # 关键专有名词的"唯一正确写法"（防同一名词多种译法）
    - term:  ""     # 例：漂泊者 / Rover —— 钉死中文主名 + 英文 + 别名
  localization_notes: # 已知的本地化风险（文化梗、双关、敬语层级…）

adaptation:
  primary_game_type:  # RPG / 开放世界 / 视觉小说 / Roguelike / 二游抽卡 …
  content_model:      # 单作完结 / 长线连载 / 章节制
  narrative_agency:   # 玩家能动性强度（强分支 / 弱分支 / 纯线性）
  protagonist_model:  # 见 §4 的 6 种玩家身份模型

north_star:
  one_sentence_concept: # 一句话能讲清这个游戏
  core_emotion:         # 想让玩家带走的那一种情绪
  core_question:        # 整个故事在拷问的那个问题
  pillars: []           # 2–4 根支柱（不可妥协的体验承诺）

canon:
  confirmed: []   # Canon：已定稿，后续必须遵守
  pending: []     # Pending：待确认，可继续探索
  rejected: []    # Rejected：已否决，别反复提
  idea_bank: []   # Idea Bank：备选灵感，暂不进主线

world:            # 见 ref 03 世界观 6 字段画布；此处只存"已 Canon 化"的世界事实

characters: []    # 每个主要角色一条；详细档案见 ref 02 人物档案模板

structure:
  mainline: []    # 主线分章 / 分幕骨架
  side: []        # 支线 / 活动
  branches: []    # 分支点与旗标（branching map，见 §8）
  endings: []     # 结局清单（若有分支结局）

production:
  deliverables: []  # 已交付 / 待交付的文档（见 §8 的 12 项清单）
  line_budget:      # 文本量预算（字数 / 词数，校准用，见 §10）
  voice_budget:     # 语音量预算（哪些角色 / 哪些场景配音）
  qa_risks: []      # 已知的连续性 / 设定风险（见 §8 Continuity Log）

live_ops:           # 仅长线运营项目需要
  version_roadmap: []  # 版本路线图（见 §6）
  events: []           # 活动计划（见 ref 05 §10.C / ref 06 §4）
  banners: []          # 卡池排期 + 叙事协同（见 §3）
  reruns: []           # 复刻计划
  recap_system:        # 前情提要 / 剧情回看机制（见 §5）
```

### 可复用要点

- **状态标签优先于内容本身**。一条没有 Canon / Pending / Rejected 标签的设定，等于无法管理的设定。
- **AI 维护、创作者拍板**。AI 负责"记住、检查冲突、提选项"；创作者负责"决定哪条升 Canon、哪条进 Idea Bank"。这条边界对齐 SKILL.md §8——AI 是脚手架，不替创作者拍创意板。
- **冲突一律显式上报 + 三选项**。默默采纳或默默丢弃是长线漂移的头号成因。
- **glossary 是廉价的高回报**。专有名词的"唯一正确写法"一旦钉死，能省掉后续大量"这个名字到底怎么写"的返工——尤其多语言项目。

## 2. 可抽角色价值栈(Character Value Stack) —— 角色设计 × 商业的桥

### 原理 / 为什么

二游角色不是"一个角色"，而是**同时承担四种职能、且四种职能必须指向同一处**（详见 ref 02 §6"为什么角色就是商品"）：

1. **玩法单位(gameplay unit)**——一套战斗方式 / 技能；
2. **IP 商品(IP product)**——被卖（抽卡）、被周边化、被二创；
3. **剧情驱动(narrative driver)**——在故事里推动情节；
4. **抽卡动机(pull motivation)**——让玩家愿意为 TA 花资源。

当四向**指向同一处**（如散兵：玩法是疾跑风系、商品是"流浪者"、剧情是须弥反派暗线、抽卡动机是"想要那个被写得可怜的少年"），角色就立得住；四向**打架**（强度拉满但剧情塌、或剧情高光但手感差），角色就会"塌房"或"无人问津"。本节给一个**逐层检查表**，把这座桥拆成 10 个可独立验收的问题。

### 模板 —— 10 层价值栈检查表

逐层一个问题 + 一个要交付的产出。从上往下填；填不出来的层，就是这个角色当前最薄弱的环节。

| 层 | 问题 | 产出 |
|---|---|---|
| **1 视觉锚点** | 一眼让人记住什么？（剪影 / 配色 / 武器 / 标志性表情 / 标志性动作） | 一句话的"一眼记忆点" |
| **2 身份锚点** | TA 在这个世界里是谁？（职业 / 阵营 / 一个秘密） | 身份 + 阵营 + 一个未公开的秘密 |
| **3 能力幻想** | 玩家抽到后幻想获得什么？（战斗方式 / 技能主题 / 玩起来的爽点） | 一句话的"玩法幻想" |
| **4 性格矛盾** | 矛盾在哪？（外冷内热 / 强大却脆弱 / 温柔却危险…） | 反差萌(gap moe)的"表层 × 内里"二元 |
| **5 关系钩子** | 和谁有未解决的关系？（宿敌 / 旧友 / 师徒 / 误解 / 血亲） | 1–2 条与既有角色的关系线 |
| **6 剧情功能** | 为什么 TA**必须**出现在这个版本？（线索 / 钥匙 / 反方 / 受害者 / 见证人） | 在本版本主线里的不可替代职能 |
| **7 拥有回报** | 抽到的玩家额外获得什么情感？（专属语音 / 个人剧情 / 特殊台词 / 待机互动） | "拥有者专属"的情感内容清单 |
| **8 未拥有体验** | 没抽到的玩家是否仍能理解 / 喜欢 TA？（主线表现 / 试用关 / 活动 / NPC 互动） | "非拥有者也能接触 TA"的路径 |
| **9 社区传播点** | 玩家会截图 / 剪辑 / 二创什么？（名台词 / 名场面 / 反差 / CP / 梗） | 1–2 个"可被传播"的高光点 |
| **10 长线空间** | 未来还能写什么？（过去 / 阵营 / 变体 / 觉醒 / 关系进展） | 至少 1 条"以后还能展开"的留白 |

### 可复用要点

- **第 4 层（性格矛盾）必须有伤口支撑**。反差萌不是"两个标签缝在一起"——表层与内里的矛盾必须能被一个**伤口(Wound)**解释，否则就是为搞笑而搞笑。配方与"避免玛丽苏"见 ref 02 §6。
- **第 7 层 vs 第 8 层是二游特有的双重写作责任**。买断制游戏只对"玩了的玩家"负责；二游必须**同时**对两类人负责：抽到的玩家要有专属回报（否则花了钱没"到账"），没抽到的玩家要仍能理解并喜欢这个角色（否则下次复刻没人想抽）。这两层的写作目标不同、不能用同一段内容应付。
- **第 9、10 层决定角色的长线生命力**。一个没有传播点、没有长线留白的角色，卖完这一波就死了；二游靠复刻、靠"以后还能写"延长每个角色的变现窗口。
- 这张表是 ref 02 §6（角色 craft）与 §3（登场排期）、§6（版本路线图）之间的**连接件**：craft 决定角色"好不好"，价值栈决定角色"能不能在商业机器里持续供货"。

## 3. 角色登场 × 卡池协同 + 版本故事模板

### 原理 / 为什么

ref 05 §5 已详述核心原则，这里给**可填模板**。一句话复述：**让玩家"想要"一个角色，靠的是让 TA 在故事里 matter（对剧情有真实分量）+ 有可共情的伤口(wound)+ 提前 N 个版本铺垫伏笔——不是临时硬塞卖货。** 分工是：**商业排期定"何时登场"（卡池表倒逼），写手用伏笔账本决定"如何登场"。**

> **ZZZ（绝区零）的正面节奏案例**：绝区零最成功的几次卡池联动（如 Miyabi/宫比、Caesar King）都遵循同一条节奏——**主线先让角色在高张力情节里令人印象深刻 → 玩家产生情感投入 → 卡池开放，此时"抽取"本身已带有情感意义**。反面做法是"先开卡池后补故事"：玩家在还不了解角色时就被要求做消费决策，抽完才慢慢认识 TA，情感价值是"事后补"的，转化与口碑都更差。
> ⚠️ 具体营收 / 流水数字属社区推算，非官方公开数据，本节只引用"节奏模式"这一可观察结论，不引用任何流水断言。
> 来源：https://www.hoyolab.com/article/42423996

### 模板 A —— 版本故事模板（每版至少四条线）

每个版本至少要同时推进四条线，缺一条就会让某一类玩家（剧情党 / 角色党 / 玩法党 / 追更党）落空：**① 主线推进 · ② 角色 spotlight · ③ 玩法·活动体验 · ④ 长线 Hook**。逐字段填：

```
版本号 / 主题：
版本一句话（这一版到底在讲什么 / 给什么）：
主推角色：
角色登场理由（TA 为何必须在这一版出现？呼应价值栈第 6 层）：
主线推进了哪一步（这一版把整体故事往前挪了多少）：
本期情绪基调（热血 / 治愈 / 悬疑 / 悲伤 / 荒诞 / 节庆 / 恐怖）：
新地点·新玩法：
活动结构（大活动套小活动？叙事比重 × 可重复性象限，见 ref 05 §10.C）：
中段反转：
高潮事件：
结尾 Hook（留给下一版的钩子）：
抽卡配合：
  ├ 角色池：
  ├ 武器 / 音擎池：
  ├ 试用关卡（让非拥有者体验角色手感）：
  └ 剧情展示（角色在本版剧情里的决定性场面）：
新玩家补偿（这一版对新入坑玩家是否友好 / 有无补足前情）：
老玩家伏笔回报（这一版回收了哪个之前埋的坑，见 §6 节奏）：
```

### 模板 B —— 卡池-叙事协同表（每个上线角色一行）

把"商业排期"和"叙事铺垫"放进同一张表，强制对齐：

| 字段 | 填什么 |
|---|---|
| **卡池角色** | 角色名 |
| **时间** | 版本前半 / 后半 / 周年 / 限定 / 复刻 |
| **抽卡前玩家已知什么** | 玩家在卡池开放时，对这个角色已经了解多少（理想：已有情感投入） |
| **抽卡动机** | 强度 / 颜值 / 性格 / 剧情 / 阵营归属 / CP / 收藏——主打哪一个？ |
| **剧情展示场景** | 角色在主线 / 活动里的高光场面 |
| **试用关卡叙事** | 试用关里用什么情节让玩家"先尝后买" |
| **角色任务解锁时机** | 个人剧情何时开放（卡池同期 / 滞后一版？） |
| **拥有后语音** | 拥有者专属的语音 / 互动（价值栈第 7 层） |
| **不拥有体验** | 没抽到的玩家如何仍能接触 TA（价值栈第 8 层） |
| **商业化风险** | 剧情强推（用主线硬捧人）/ 强度不符 / 限定焦虑 / 角色塌房 |
| **叙事补救** | 复刻 / 常驻化 / 补角色档案 / 后续补写——出问题怎么救 |

### 可复用要点

- **顺序铁律：先让角色 matter，再开卡池。** ZZZ 的正面案例与 ref 05 §5 的散兵案例同源——情感投入在前，消费决策在后。
- **"剧情强推"是高频商业化风险**。为了卖角色而让 TA 在主线里突然抢戏、压过主角（见 §4 主角能动性铁律），会同时伤害"被抢戏的主角"和"被硬捧的新角色"。
- **限定 / 复刻是叙事规划的一部分，不是事后补丁**。表里的"叙事补救"列要在登场时就想好——一个限定角色错过了，复刻是唯一的二次变现窗口，复刻时的剧情接触点要提前留好。

## 4. 玩家身份模型(player-avatar) + 主角能动性

### 原理 / 为什么

"玩家在故事里是谁"是二游的**地基级决策**——它决定代入感强度、世界可扩展性、商业转化方式，且一旦定下就极难改。不同模型各有优劣，没有"最好"，只有"匹配你的项目"。

### 模板 A —— 6 种玩家身份模型表

| 模型 | 特征 | 优点 | 风险 | 写作建议 |
|---|---|---|---|---|
| **指挥官 / 老师 / 舰长** | 玩家是团队核心，角色们围绕"你"运转 | 代入感强；天然解释"为何这么多角色聚在身边" | 易写成"万人迷"或纯工具人 | 给主角**明确职责和限制**，别让 TA 什么都会、人人都爱 |
| **固定主角（有名有性格）** | 主角是具体的人，有自己的台词与立场 | 剧情张力强，能写真正的人物弧 | 代入感弱（玩家未必认同 TA 的选择） | 让玩家**理解主角的选择逻辑**，即使不认同也能"看懂为什么" |
| **旅行者 / 观察者** | 主角穿梭各地区，串起世界 | 易扩展世界（换地图就换故事） | 主角像"摄像机"，缺乏自身压力 | **每章给主角一个个人压力**，别让 TA 只是路过看戏 |
| **群像 / 无唯一主角** | 多视角叙事，没有单一中心 | 世界厚重、信息密度高 | 玩家缺乏情感落点，容易"不知道该关心谁" | **每版设一个 POV 锚点角色**，给玩家本期的"代入对象" |
| **玩家捏人 Avatar** | 高自定义，玩家塑造自己的角色 | 自由度高、代入感强 | 主角没固定性格，剧情个性难写 | **用外部角色（NPC / 同伴）投射主题**，主角留白、配角承重 |
| **关系陪伴型主人公** | 与角色建立亲密 / 恋爱关系 | 商业转化强（情感付费） | 边界争议风险高（人设承诺 / 玩家期待落差） | **明确关系基调**，别乱开空头承诺、别让边界模糊引发舆情 |

### 模板 B —— 主角能动性铁律（来自鸣潮 CBT / 1.0 翻车）

> 鸣潮（Wuthering Waves）CBT 与 1.0 开服因"沉默 / 工具型主角"遭遇大规模批评，是研究"主角能动性"的最佳负面案例（开场翻车的完整诊断见 ref 05 §2.5）。官方后续公开承认重写了约 90% 的剧情。
> ⚠️ 部分细节（"阿呜事件"等具体场景）来自玩家社区回忆，非逐字官方记录；此处提取的是**可复用的设计原则**，非剧情事实断言。
> 来源：https://insider-gaming.com/wuthering-waves-rewrites-90-of-story-devs-issue-apology

沉默 / 工具型主角的**三大致命陷阱**及对应原则：

1. **NPC 在场景中主动稀释主角的成就**
   - 翻车表现（"阿呜事件"）：玩家辛苦打赢 BOSS，叙事焦点却立刻切到某个 NPC"多年的愿望终于实现"，主角沦为背景板，玩家的努力没有被故事承认。
   - **原则**：玩家行动的**直接后果**必须在**下一个叙事节拍**就被世界确认——先认账给玩家，再去分配 NPC 的情绪戏。

2. **NPC 态度没有成本锚点**
   - 翻车表现：CBT 全程对主角莫名敌视 ↔ 1.0 全城对主角即时热情狂吹——**两极都失真**，因为认可来得没有代价。
   - **原则**：**认可应当以行动为代价换取**。设计明确的"态度转变节点"（主角做了某件具体的事 → 某个 NPC 的态度因此改变），别让好感凭空发生或凭空消失。

3. **主角是"信息中转站"而非"行动者"**
   - 翻车表现：主角只负责听 NPC 解释设定、跑腿传话，不做有分量的决定。
   - **原则**：关键节点的**主角选择要有可见的近期后果**——哪怕分支最终殊途同归，玩家选 A 与选 B 在**语气、NPC 当场反应、短期演出**上也要有差异，让选择"有重量"。

### 可复用要点

- **身份模型是地基，定了就别轻易改**。改主角身份模型 = 重做代入感地基，代价极高（鸣潮重写 90% 就是代价）。
- **沉默主角不等于"没存在感"**。沉默是为代入服务的设计选择，但必须配"嘴替 / 搭档"补足表达（ref 05 §7.2 的派蒙模型），且世界要持续"认账"主角的行动。
- **"关系陪伴型"商业转化最强、舆情风险也最高**。基调要在企划期就明确，写作时严守边界——这类项目的塌房往往不是剧情烂，而是"人设承诺被违背"。

## 5. 可跳过但不空洞(Skip-Resilient Writing)

### 原理 / 为什么

二游玩家是**分层**的：有人全读、有人全跳、有人只看角色相关、有人只想打玩法。**剧情跳过(skip)已是行业标配**——围绕"该不该让玩家跳剧情"的争论早已尘埃落定，但它折射出一个真实压力：**玩家对冗长、低密度剧本的不耐烦是常态**。所以二游剧本要写成**可跳过但不空洞**——既要让跳过的玩家不丢失关键信息，又不能因为"反正能跳"就把内容写烂。

> 来源：https://36kr.com/p/3165149711510274 （游戏葡萄：剧情跳过争论的尘埃落定与玩家对冗长剧本的不耐烦）

### 模板 —— 三层写作模型 + 检查清单

把每一段剧情按"三层信息"组织，确保不同深度的玩家各取所需：

| 层 | 服务谁 | 这一层要传达 |
|---|---|---|
| **直觉层(intuitive)** | 跳过 / 半跳过的玩家 | 角色印象、当前目标、情绪基调、下一步要去哪 |
| **逻辑层(logical)** | 正常阅读的玩家 | 因果链、反转、动机、关系变化 |
| **深层表达(deep)** | 深度玩家 / 考据党 | 主题、伏笔、隐喻、长期谜团 |

**交付前检查清单（逐条问）**：
- [ ] 开场 **30 秒内**是否有一个冲突 / 钩子？（别让前 30 秒全是设定铺陈）
- [ ] 每 **2–3 分钟**是否有一次"状态变化"（情绪 / 处境 / 信息）？
- [ ] 重要信息是否有**两种载体**承载？（如：对白说一遍 + 任务目标摘要再写一遍，跳过对白的玩家仍能 get）
- [ ] 是否有**前情提要(recap)**和**任务目标摘要**？（断更几周回来的玩家能不能立刻接上）
- [ ] 关键剧情是否能在**日志 / 剧情回看**里重读？（错过 / 跳过的玩家有没有补救入口）

### 可复用要点

- ⚠️ **关键：跳过友好 ≠ 可以写烂。** 恰恰相反——当玩家能自由跳过，"真正留下来看的内容"承受的质量期待反而更高。跳过功能筛掉了不耐烦的玩家，留下的是愿意投入的核心受众，对他们糊弄不得。
- **"两种载体"是对抗跳过的最实用招式**。把关键信息同时编码进"对白"和"可被快速扫到的载体"（目标摘要 / NPC 一句总结 / 任务标题），跳过对白的玩家不会卡在"不知道现在要干嘛"。这也呼应 ref 05 §8.1 对崩铁"对话仅限一次、无回看、点快了直接错过"的批评。
- **recap 系统是长线运营的刚需**，不是可选项——见 §6 路线图与 §8 交付清单里的 Recap/Archive Plan。

## 6. 长线版本路线图 + 伏笔节奏

### 原理 / 为什么

长线连载的"连贯"不靠灵感，靠**节奏管理**。玩家对二游的信任，是建立在"埋的坑会被按节奏回收"之上的——回收太慢累积成不信任（"故弄玄虚"），回收太快又会让世界显得单薄。需要一份**版本路线图**把主线推进、角色登场、伏笔回收、情绪峰值排进时间轴。

### 模板 —— 版本路线图（可填表，给 1.0–1.3 留空行）

| 版本 | 主线推进 | 主推角色 | 阵营 / 地区 | 活动类型 | 情绪基调 | 新玩法 | 伏笔回收 | 下版 Hook |
|---|---|---|---|---|---|---|---|---|
| 1.0 |  |  |  |  |  |  |  |  |
| 1.1 |  |  |  |  |  |  |  |  |
| 1.2 |  |  |  |  |  |  |  |  |
| 1.3 |  |  |  |  |  |  |  |  |

**填表原则**：
- **1.0 建立核心体验**——把"这游戏到底好玩 / 好看在哪"立住，别在首发版本就埋一堆看不懂的坑。
- **每版既有本期满足也有长期未解**——本版要给一个闭合的"到账"（情绪 / 信息回报），同时留一个更大的未解（追更动力）。这是 ref 05 §2.1/§2.4 反复出现的"双重收尾"。
- **角色池要有登场理由**——每个上线角色都对应版本故事模板（§3）里的"角色登场理由"，不临时硬塞。
- **每 3–4 个版本回收一个中型伏笔**——信任建立在节奏上。太久不回收，玩家会怀疑你"根本没想好"。
- **周年(anniversary)承担情绪峰值或世界推进**——周年是全年最重的节点，要么给一次情绪大爆发，要么推动核心世界观一大步。
- **复刻(rerun)和常驻化进入叙事规划**——不是运营单独的事，写手要知道哪个角色会复刻、复刻时给什么剧情接触点（呼应 §3 卡池协同表的"叙事补救"列）。

### 伏笔节奏的反面教材（交叉引用）

- **"一章塞太满 → 草率收场"**：战双帕弥什把需要 N 章的故事硬压进 1 章，结尾必然靠"嘴炮"强行收束（详见 ref 05 §2.7）。伏笔节奏不只是"回收够不够快"，也是"单章别贪多"。
- **"长期大谜团"的反面警示 Amphoreus（翁法罗斯）**：崩铁这条横跨 8 个版本（2025，3.0–3.7）的行星弧，恰恰是**伏笔节奏失控**的负面案例——同时开太多坑、版本结束收束太少，玩家陷入"下个伏笔来了上个还没开"的累积焦虑。**伏笔节奏的完整工艺（公平伏笔 / 谜题箱陷阱 / "打开 3 个关 2 个 > 打开 10 个关 0 个"）见 ref 10 PART 四·谜**；行星=幕、各幕独立 controlling idea 的结构面见 **ref 01 §1** 与 **ref 05 §2.2**。
- **伏笔账本**：把每个坑登记"在哪一章/版本回收、只有总编辑能改"——这是 ref 05 §1.3 的核心防崩工具，路线图的"伏笔回收"列就是伏笔账本在时间轴上的投影。

### 可复用要点

- **路线图是"活文档"，进 Project State 的 `live_ops.version_roadmap`**（§1）。它会随创作不断改，但每次改都要检查"伏笔回收节奏"没被打乱。
- **节奏 > 数量**。玩家不会数你埋了多少坑，但会强烈感知"坑回收得勤不勤"。每 3–4 版一个中型回收，是维持信任的经验节律。

## 7. 演出(presentation) scope 管理 —— 给小团队的现实主义

### 原理 / 为什么

2024–2026 年，二游的"演出(presentation)军备竞赛"持续升级，几条新基准线已经形成：

- **动态立绘 / 微表情成新基准线**——纯静态立绘 + 文字 ADV 已被部分玩家视为"低成本信号"；
- **战斗与剧情的连续性**——高潮战斗里内嵌 CG、战斗即演出；
- **音乐叙事(music as narrative)**——配乐的时机与剧情节拍精确对齐，而非背景音；
- 海外 reaction 主播的传播，使"能被 reaction 的演出片段"本身成为一种设计目标。

> 来源：https://www.thepaper.cn/newsDetail_forward_30869190

问题在于：**军备竞赛是头部厂商用钱堆出来的，小团队照抄必死。** 本节给小团队的不是"怎么追上军备竞赛"，而是"怎么在有限预算下，让演出仍然有效"。

### 模板 —— 小团队演出铁律（避免陷阱）

1. **承认限制，聚焦剧情密度而非演出规模。**
   - 范例：**重返未来1999(Reverse:1999)** 以**风格化的静态 ADV + 高密度角色对白**建立口碑，而非堆 3D CG——它把差异化压在"译制片腔 / 年代感 / 文学性"上（详见 ref 05 §2.6）。证明"演出规模不够，可以用剧情密度和腔调补"。

2. **把演出预算集中在 2–3 个关键情绪节点。**
   - 别把有限的 CG / 全语音 / 特殊镜头**平均分配**——平均分配的结果是"全部平庸"。挑出整段剧情里最该爆发的 2–3 个情绪顶点，把演出资源砸进去；其余用文字 + 立绘扎实托住。

3. **文字本身可以是演出。**
   - 好的感官细节、潜台词(subtext)、准确的情绪节奏，比一段廉价 CG 更有传播力。一句对得准的台词会被玩家截图传播，一段画质粗糙的 CG 不会。**"演出"不只指视听资源，也指文字的精准度。**

### 可复用要点

- **演出意识 ≠ 演出预算**。写场景时标注"这句配什么表情 / 镜头 / BGM"是写作纪律（ref 05 §7.3、ref 08 演出层），但**标注不等于必须做满**——标注的价值在于让你知道"哪 2–3 个点最值得花钱"。
- **静态对话墙是流失重灾区**（ref 05 §8.1）。预算有限时，对策不是"全加 CG"，而是"砍掉无演出的长对话，把省下的资源集中到关键节点"。
- 演出层的具体脚本格式（立绘 / 表情差分 / 背景 / BGM / 语音标注）见 **ref 08**。

## 8. 制作级文档打包(production handoff) —— 团队 / 外包可用的交付物

### 原理 / 为什么

当项目从"一个人写"走向"多人 / 外包协作"，散落的笔记必须**打包成标准交付物**，否则新加入的写手 / 美术 / 程序 / 本地化无法对齐。这是 ref 05 §1.3"章节负责人对单章全责（文本+角色+演出）"在文档层面的落地——每个职能都要能从一份文档拿到 TA 需要的东西。

### 模板 —— 12 项交付清单

| # | 交付物 | 内容 | 给谁用 |
|---|---|---|---|
| 1 | **Creative Brief（创意简报）** | 一句话概念 + 核心情绪 + 支柱（north_star 的对外版本） | 全员对齐 |
| 2 | **Story Bible Lite（故事圣经简版）** | 主题、基调、世界一句话、主线大纲——给"不需要读全部设定"的人 | 美术 / 市场 / 外包 |
| 3 | **World Bible（世界观文档）** | 完整世界设定（已 Canon 化部分） | 设定 / 编剧 |
| 4 | **Character Bible（角色圣经）** | 每个角色的档案（ref 02 模板）+ 价值栈（§2） | 编剧 / 美术 / 声优指导 |
| 5 | **Mainline Beat Sheet（主线节拍表）** | 主线逐节拍骨架（ref 01 框架锚点） | 编剧 |
| 6 | **Quest Docs（任务文档）** | 单个任务的流程 + 目标 + 触发 | 关卡 / 程序 / 编剧 |
| 7 | **Branching Map（分支图）** | 分支点、旗标(flag)、结局走向（见 ref 08 §3） | 编剧 / 程序 |
| 8 | **Dialogue Sheets（对白表）** | 逐行对白 + 说话人 + 演出标注（格式见 ref 08 §2） | 编剧 / 本地化 / 配音 |
| 9 | **Terminology Glossary（术语表）** | 专有名词唯一写法 + 译法（= Project State 的 glossary） | 本地化 / 全员 |
| 10 | **Continuity Log（连续性记录）** | 已确认事实 + 已知矛盾 + 待解风险（= qa_risks） | 总编辑 / QA |
| 11 | **Narrative QA Checklist（叙事 QA 清单）** | 交付前过的反 AI 味 / 反套路 / 连续性检查（基于 ref 09） | QA / 总编辑 |
| 12 | **（Live Service 额外）** | Version Roadmap（§6）+ Event Plan（ref 05 §10.C）+ Banner Narrative Plan（§3）+ Recap/Archive Plan（§5） | 运营 / 编剧 |

### 可复用要点

- **不是每个项目都要 12 项全做**——单人项目可能只需要 1/2/4/5/8/9。清单的价值是"提醒你哪些维度可能漏了"，按项目规模裁剪。
- **对白表 / 演出脚本的具体格式见 ref 08**（Format A 剧本式 / Format B 对白表 / Format C 注解散文体 + ADV 演出层模板）。本节只列"要交付什么"，ref 08 管"长什么样"。
- **第 9、10 项（术语表 + 连续性记录）就是 Project State 的导出视图**（§1）——别维护两份，从活文档生成交付物。

## 9. 工作模式选择器(working-mode) —— 让创作者选 AI 的介入深度

### 原理 / 为什么

SKILL.md §2 的**水平校准(L0–L3)**是 **AI 主动评估**创作者的写作水平、据此调辅导力度。本节的**工作模式**是它的**补充**：让**创作者主动选择**希望 AI 介入到多深。两者**正交**——校准回答"这个人写作水平如何"，工作模式回答"这个人现在想要 AI 帮多少"。

**为什么要正交**：一个 L0 新手也可能选 D（"我现在没空学，先给我一版初稿"）；一个 L3 老手也可能选 A（"我只想要思路，字我自己写"）。把"水平"和"想要的协作深度"绑死，会误判需求。

### 模板 —— 5 档工作模式（A–E）

| 档 | 模式 | AI 做什么 |
|---|---|---|
| **A** | 只引导思考，不替定稿 | 苏格拉底式提问、给框架、给检查清单；**不产出成稿文字** |
| **B** | 给多个方案，我来选 | 针对一个点给 2–3 个不同方向，创作者挑 |
| **C** | 推荐主方案并说明理由 | 给一个推荐方案 + 为什么，创作者拍板或调整 |
| **D** | 直接产出可用初稿，我再改 | 写出可用的初稿文字，创作者在上面修改 |
| **E** | 像叙事总监一样管理全流程 | 主动维护 Project State、排路线图、提醒节奏、统筹交付 |

### 可复用要点

- **与水平校准(L0–L3)正交**，别混为一谈。校准调"教学姿态"（多少术语、多少示范），工作模式调"产出深度"（给思路还是给成稿）。
- **AI 开场可顺带一问**——在 S0 校准之后，可以轻描淡写补一句"你希望我更多是帮你想（给思路）、还是帮你写（给初稿）？"，把工作模式定下来。但**不要做成问卷**，对话式带过即可（呼应 SKILL.md §2.2"绝不像考试"）。
- **模式可随时切换**。同一个项目里，构思阶段选 A/B、卡壳赶进度时切 D，都正常。AI 应跟随，不锁死。

## 10. 长度基准(calibrate ambition) —— 别把新人吓死，也别让其低估工作量

### 原理 / 为什么

新人对"二游剧情要写多少字"几乎没有概念——要么野心过大（"我要一上来铺满整个世界"），要么严重低估（"写个几千字主线应该够了"）。给几个**可引用的真实基准**，帮创作者校准野心 vs 现实。

> ⚠️ **以下数字均随版本更新而变化，且部分来自社区 / 聚合统计而非官方逐字数据。** 引用时一律带"随版本更新 / 社区统计"的限定，不作为精确断言。

### 模板 —— 头部二游 / 主机叙事的体量基准

- **明日方舟 Episode 08 单章约 140,843 中文字 / 78,010 英文词**。官方直播（海猫 / 阿夹 livestream）称这个体量足以填满"两本电影剧本"。
  ⚠️ 数字来自官方直播口径，单章体量随章节差异很大。
- **FGO Lostbelt 7 仅"阅读时间"约 11 小时**（Part 1 约 4 小时 28 分 + Part 2 约 6 小时 29 分）——这还只是"读完"的时间，不含战斗。
  ⚠️ 数据来自 Atlas Academy DB（社区数据库）的阅读时长统计。
- **蔚蓝档案(Blue Archive)每章约 16–27 话**——单章由十几到二十几个"话"组成，是"短话 × 多话"的结构。
  ⚠️ 各 Volume / 活动差异较大。
- **Persona 5 Royal 主线平均约 106 小时 + 23 个 Confidant（社群链），每个最多 10 阶**——这是"主线 + 大量一对一关系剧情"的总体量。
  ⚠️ 时长来自 HowLongToBeat 玩家聚合统计，因人而异。

### 给新人的结论

- **头部二游的单章体量是数万到十几万字**——这是几人到十几人编剧团队、按 42 天周期持续产出的结果，**不是一个新人一次性该对标的目标**。
- **新人 / 小团队的正确起点是"垂直切片(vertical slice)"**：先做**一个篇章 + 一个角色 + 一个场景**的完整切片，把这一小块从世界观到对白到演出全部打磨好，而不是铺满整个世界、然后处处半成品。
  - 这条与 **SKILL.md §10 整体完成判据**一致：先拥有"一段写好并改过的具体场景脚本（有目标+灾难、潜台词、可分辨角色声音；二游含演出层）"，比拥有一堆没写透的设定更有价值。
  - 也对齐 ref 03 的"世界观拖延症 / 自嗨"警告：背景笔记字数超过正文就是病。

### 可复用要点

- **用基准做"现实锚"，不做"必达 KPI"**。告诉新人"头部是这个量级"是为了让 TA 校准期待，不是要求 TA 第一周写十万字。
- **垂直切片是新人唯一靠谱的起点**。一个打磨透的小切片，既能验证创作者的能力、又能产出一份真正能带走的样章；铺满全世界的半成品则两头落空。
- 数字会过时——引用时永远带 ⚠️ 限定，且鼓励创作者自己去对应游戏的 wiki / DB 查最新数据。
- 来源汇总见文末 Sources §10（compass research：Arknights livestream 海猫/阿夹；Atlas Academy DB；HowLongToBeat via SVG）。

## 一句话速记

- **§1 Canon 管理**：长线创作靠一份活文档当单一事实来源，给每条设定打 Canon / Pending / Rejected / Idea Bank 标签，新想法与 Canon 冲突时 AI 必须显式上报 + 给"改新想法 / 改旧设定 / 变成剧情冲突"三选项。
- **§2 角色价值栈**：二游角色同时是玩法单位 + IP 商品 + 剧情驱动 + 抽卡动机，四向须指向同一处；用 10 层检查表逐层验收，性格矛盾必须有伤口支撑，"拥有回报 vs 未拥有体验"是二游特有的双重责任。
- **§3 登场协同**：先让角色在故事里 matter、玩家有情感投入，再开卡池（ZZZ 正面节奏）；商业排期定"何时登场"，伏笔账本定"如何登场"。
- **§4 玩家身份**：6 种身份模型各有优劣、定了别轻改；主角能动性三铁律（玩家行动当场被世界认账 / 认可须以行动换取 / 关键选择有可见近期后果）来自鸣潮重写 90% 的教训。
- **§5 可跳过不空洞**：按直觉层 / 逻辑层 / 深层三层写，关键信息给两种载体；跳过友好绝不等于可以写烂，反而抬高了核心内容的质量期待。
- **§6 版本路线图**：每版"本期满足 + 长期未解"，每 3–4 版回收一个中型伏笔（信任建在节奏上），周年承担情绪峰值，复刻进规划。
- **§7 演出 scope**：小团队承认限制、把演出预算集中到 2–3 个情绪顶点、其余靠剧情密度——文字本身就是演出（重返未来1999 范例）。
- **§8 制作打包**：12 项交付清单按项目规模裁剪，术语表 + 连续性记录直接从 Project State 导出，对白/演出格式见 ref 08。
- **§9 工作模式**：A–E 五档让创作者选 AI 介入深度，与 L0–L3 水平校准正交（新手也可能要初稿，老手也可能只要思路）。
- **§10 长度基准**：头部二游单章数万到十几万字（⚠️ 随版本/社区统计），但新人/小团队的正确起点是"一篇章 + 一角色 + 一场景"的垂直切片，而非铺满世界。
