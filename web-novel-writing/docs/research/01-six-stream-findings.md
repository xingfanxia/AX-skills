## 调研流：网文创作的工业方法论 + 平台机制

**摘要**：网文长篇连载是一套"数据驱动的工业流程"，不是灵感写作：作者用"粗纲(一句话)→细纲(场景/幕)"或产品化的"总纲→卷纲→章节蓝图"三级大纲，先锁定核心冲突与结局再倒推大事件串，并靠"设定集+人物小传+世界观文档+伏笔追踪表+人物关系图+时间线"做人工state管理来防止自己写崩。平台机制把这套流程量化成可监控指标：番茄看三章追读率(≥35%及格/≥45%优秀)、完读率、追更率(>40%流量倾斜5倍)，起点看首订/均订，黄金三章/章末钩子/爽点密度直接决定留存与收入。爽点本质是"安全感的社会延伸"(尊重感→打脸、优越感→装逼)加上"期待感"这一条件反射，核心机制是"压抑-释放/情绪守恒"——憋屈时长必须与爽感强度成正比；毒点(绿帽、送女、降智、圣母、注水、断更/太监)是会一票否决的劝退雷区。对AI pipeline最关键的结论：一线作者与工程复盘一致证明，AI在"环境描写/旁白/扩写/前情总结/命名"上可自动，但"剧情走向/节奏拍板/情绪设计/打动人心"必须人类主导；长篇崩坏的根因是"AI自我一致性随篇幅衰减"(贵模型解决不了)，唯一可行解是把流程原子化拆分、用确定性代码强制约束(知识图谱一致性校验/伏笔台账/正典事实分级)、并让独立agent做外部验证而非自审。


### 关键发现

- **[high]** 网文大纲是分层结构，最小可执行单元是'幕/场景'，应做成填表式模板而非自由散文：粗纲=一句话(给编辑/自己看)，细纲=场景级分镜稿，每个场景固定填写'渴望/障碍/冲突/明面行动/情感/展示(人设)/爽点'七要素。
  - 依据：大神小密圈细纲模板原文：'渴望:保护老师→障碍:身份/性格/年龄差→冲突:莫北VS罗凯泽→行动:黑账号匿名威胁失败后发匿名信→情感:正义感→展示:身份/技术→爽点:扒出情敌是人渣'；知乎专栏'网文细纲详解'明确'细纲基础单元是幕，一幕就是一个场景，含信息/形象等五类要素'。月入万元作者强调'最好用的大纲是填表，越简单越好用'。
  - 出处：https://aitool.csdn.net/671a0fface231628fb71e5e5.html ; https://zhuanlan.zhihu.com/p/692322925 ; https://write.qq.com/ask/qamenvx
  - 启示：AI pipeline应把'章节蓝图'定义成结构化schema(场景列表，每场景含目的/冲突/行动/爽点/钩子字段)，让AI填表而非自由发挥；这天然把'给AI的约束'与'正文'分离，是解决prompt泄漏到正文的关键——约束放在结构化字段里，正文生成只消费字段。
- **[high]** 专业方法是'先有结局/核心冲突再倒推'：先定主角从出生到巅峰的大事件串(几百到几千字主线)，再逐卷细化；场景级用'目标导向+反推法'(先定场景目的→反推不可回头的关键行动→安排转变点)。30万字处(≈上架节点)要埋长期主线锚点绑定读者期待。
  - 依据：五步大神大纲法:'先设定一个笼统大纲，一条清晰主线从主角出生到结束，再将每个大事件分卷细化'；反推写作法'你必须知道这场戏达成什么:推进剧情/揭露角色/建立情感/制造冲突，再反推关键行动(谎言/告白/揍人/拿出证据)'；20条黄金法则'30万字设长期主线锚点(飞升之约/灭族之仇)，30万字≈上架节点，大高潮显著提升订阅率'；B站有'新人大纲30万轻松写锚点倒推法'。
  - 出处：https://vocus.cc/article/62eff7dafd89780001e5cc41 ; https://vocus.cc/article/6852cb71fd89780001e025ea ; https://www.bilibili.com/opus/1029568922899185669
  - 启示：pipeline的第一步必须是人类/人机共定的'结局+主线锚点+核心冲突'，作为不可变的顶层契约(canon)，所有后续章节生成都受其约束；AI禁止自行改写顶层锚点，只能在卷纲/章纲层填充——这正是防止'4-5章后世界观崩'的根。
- **[high]** 黄金三章是平台差异化的硬留存机制，且各平台节奏阈值不同：番茄首章1500-2000字、冲突≤500字、金手指≤1000字、考核完读率/留存；起点2000-3000字、可慢热≤1000字、金手指第三章前、考核世界观/升级线/人设；飞卢日更6000字、开篇300字内必须有爆点。功能分工:第1章立困境、第2章兑现卖点(金手指)、第3章扩大冲突+钩子。
  - 依据：8大平台开篇节奏对照表(番茄/七猫/起点/晋江/飞卢/纵横/17K/书旗各列首章字数/铺垫容忍/冲突时限/金手指时限/考核重点)；阅文编辑'夜宵'用'耐心值理论'解释:书名/简介/题材决定第一印象→决定你有黄金多少章；编辑共识'前5000字未出现核心冲突容易被淘汰，晋江签约规则要求前三章完成世界观铺设+主线悬念'。注意趋势'黄金三章→黄金三句'。
  - 出处：https://wangwen666.com/post/204.html ; https://m.write.qq.com/portal/article/detail?CAID=12871234803277101 ; https://maliangwriter.com/guides/fanqie-novel-signing-guide
  - 启示：pipeline必须参数化'目标平台'，把开篇节奏阈值(冲突/金手指出现的字数位置)作为可校验的硬规则注入开篇生成，并能用规则扫描器自动检测'第N字前是否出现冲突/金手指'。
- **[high]** 平台用可量化留存指标决定流量分配，追读率(追更/阅读比)是生死线、完读率只决定收入高低：番茄三章追读率≥35%及格/≥45%优秀，完读率(10万字)≥15%/≥20%，追更率≥40%可获约5倍流量倾斜；番茄给量是14天周期、'没有首秀一说'；起点看首订(过万=爆款，均订过万约900本但首订过万仅百余本)。
  - 依据：龙空/起点社区内部资料帖:'追更率才是一切…完读率不是一切，完读高只是钱多…评分根据吸量/留存/完读综合判断，影响曝光次数'；snowflake-fiction项目给出'番茄平台数据指标标准'表(追读率/完读率/追更率的及格线/优秀线/影响)；起点万订书单'首订=作品首个VIP章24小时内订阅>1W'。
  - 出处：https://www.qidianclub.com/threads/428 ; https://github.com/hestudy/snowflake-fiction ; https://www.qidian.com/booklist/detail/643544
  - 启示：pipeline应内置'数据反馈闭环'：监控追读率/完读率→定位高跳出章节→触发该章节重写。但要注意:这些阈值是经验/内部资料推断(平台不公开算法)，应作为可调参数而非硬编码常量。
- **[high]** 爽点的底层机制可工程化:爽点=安全感的社会延伸(尊重感→打脸、优越感→装逼、认同感)，'期待感'是独立的条件反射式爽点；核心节奏是'压抑-释放/情绪守恒'——憋屈时长必须与爽感强度成正比，且'先打一巴掌再给甜枣'(先降期待再超预期解决)。
  - 依据：龙空精华帖'对爽点的系统分析':'尊重感代表地位→打脸需要铺垫先有需求；优越感→装逼，对比角色越强越多效果越好；期待感来源于巴甫洛夫条件反射'；新媒体爽点教程把装逼分三层(基础装/温酒斩华雄式侧面压/狠压式)并提'先打一巴掌再给甜枣'；爽文教程提'情绪守恒定律:憋屈时长需与打脸力度成正比'。
  - 出处：https://www.qidianclub.com/threads/1326 ; https://wangwen666.com/post/44.html ; https://vocus.cc/article/685aaa33fd8978000172d736
  - 启示：爽点不是'写好人物'的玄学，而是有可编码的因果链(需求建立→压抑→释放)。pipeline可把'情绪债账本'做成状态:记录每条憋屈的强度与时长，强制要求对应释放强度≥债务，自动检测'虐点无补偿/爽点免费(无铺垫)'。
- **[medium]** 爽点密度有可量化曲线:每章≥1个小爽点、每3-5章1个中爽点、每卷1个大爽点；追读率top10%作品平均每1.8章一次情绪高峰，bottom10%是4.7章，'平路'超过3章追读开始掉。snowflake给出更粗的工业刻度:每3章小爽/每10章中爽/每30章大爽(阶段Boss)/每100章超大爽(换地图)。
  - 依据：马良写作节奏文:'我们分析了追读率top10%作品，平均每1.8章有一次情绪高峰，bottom10%平均间隔4.7章…平路走超过3章追读就开始掉'；snowflake-fiction爽点密度表;林海听涛(竞技文大神)'升级是小爽点不要太多笔墨，读者期待的是升级后打脸；打脸要拉长高潮，用围观者评头论足充分利用高潮'。
  - 出处：https://maliangwriter.com/blog/novel-pacing-hook-design-guide ; https://github.com/hestudy/snowflake-fiction ; https://ady101.com/ys/xtjc/nanp/chm/000021/6192.htm
  - 启示：pipeline可把'爽点密度'做成可校验约束(每章蓝图必须标注本章爽点类型与级别)，并跨章统计密度，自动预警'连续N章无情绪高峰'。1.8/4.7是单一来源数据，标medium。
- **[high]** 章末钩子是追读命门，有4种可模板化类型(悬念式/反转式/情绪炸弹式/信息投放式)，且必须在章节蓝图里提前规划(蓝图最后一个字段=钩子)，不能写完再想结尾。章节名本身是'读者承诺层'，要既提高续读意愿又被本章内容兑现。
  - 依据：马良写作:'读者决定要不要看下一章就看你这章最后三行'，列出四类钩子模板并强调'钩子要在蓝图里提前规划'；webnovel-handbook把'章末钩子'升级为跨章'读者问题链/好奇心缺口'管理(本章主问题/旧问题推进/新问题/答案利息/假钩子识别)，并单列'章节名=读者承诺层'专项。
  - 出处：https://maliangwriter.com/blog/novel-pacing-hook-design-guide ; https://github.com/miserylee/webnovel-handbook
  - 启示：钩子是结构化字段，AI最适合做的就是'按蓝图钩子字段生成最后三行'。pipeline应把'本章主问题/承接上章悬念/新增悬念'作为显式状态跨章传递，避免AI忘记前面挖的坑。
- **[high]** 毒点是会一票否决的劝退清单，必须用规则扫描而非靠感觉:男频十大毒点(绿帽#1/主角弱智/反派降智/圣母/重要人物死亡/反三观/水文/断更/烂尾/太监)；五大类(人设:圣母白莲花/双商掉线/油腻种马；逻辑:降智反派/金手指崩坏/重复套路；三观:绿帽/送女/夹带私货；文笔:注水/对话尴尬/视角混乱；题材:历史硬伤/专业过家家)。平台容忍度不同(飞卢对绿帽极敏感，晋江三观审核最严)。
  - 依据：搜狐'网络小说十大毒点'与网文666'五大雷区'两份独立清单高度一致；webnovel-handbook专设'读者弃书触发器/毒点雷点/留存安全'文档，要求AI先判断目标读者/类型承诺/触发强度/出现位置/铺垫预警/主角主动性/情绪补偿，再决定避开/预警/改写/缩短/补偿/保留；实操建议'每写5万字回头倒查:主角动机/反派智商/金手指规则前后矛盾'。
  - 出处：https://www.sohu.com/a/234316624_100113218 ; https://wangwen666.com/post/194.html ; https://github.com/miserylee/webnovel-handbook
  - 启示：毒点清单是pipeline最容易自动化的安全网:做成确定性规则扫描器(检测绿帽/送女/降智/视角混乱等模式)+按平台切换容忍度，作为'章节发布前体检'的硬gate。
- **[high]** 长线一致性靠人工'state账本':伏笔有6类(身份/语言/场景/数字/主题/物件)，核心工具是'伏笔追踪表'(每条记录:出现章节/类型/回收计划章节/状态[已埋/已微回应/已回收])；外加设定集/人物小传/人物关系图/时间线。铁律:'别把伏笔当钩子用'(埋时无需读者立刻注意)、'埋了忘回收=烂尾'。
  - 依据：网文666'6种伏笔类型拆解'给出追踪表三建议:'建一张伏笔追踪表记录出现章节/类型/回收计划/状态…很多作者写着写着就忘了三个月前埋的伏笔'；山音编剧master列'埋种/回调登记表逐条审计'为反烂尾手段；鱼人二代《校花》2201万字仍'有很多伏笔要一一填坑'印证账本是长线刚需。
  - 出处：https://wangwen666.com/post/200.html ; https://github.com/Shanyin-ai/shanyin-screenwriting-master ; https://www.jfdaily.com/wx/detail.do?id=593182
  - 启示：伏笔追踪表是pipeline里最纯粹可自动化的部分:做成机械化的台账(埋设时自动登记、状态机流转、临近回收章自动提醒)。但'伏笔设计'(埋什么/何时回收)是人/半自动决策。
- **[high]** 长篇崩坏的根因是'AI自我一致性随篇幅衰减'(内在衰减问题，不是模型质量问题，用更贵模型解决不了)，且'AI同轮扮演所有角色自审'必然失败——审核agent其实就是刚才的执笔agent，不会认真挑自己的错(自欺)。
  - 依据：腾讯云8天工程复盘'AI长篇质量保证体系':'最大的坑不是写不好，而是写得多错得多…AI一次性生成大量内容时自我一致性随篇幅下降，前面埋的伏笔后面忘了，前面的风格后面崩了…这是AI生成的内在衰减问题，用更贵的模型解决不了'；'角色分工只是表面，执行者没变，C不会挑B的错因为等于承认自己写得有问题'。
  - 出处：https://cloud.tencent.com/developer/article/2650227
  - 启示：直接印证用户的'4-5章后崩'约束。pipeline设计必须:(1)限制单次生成长度(场景切片)，(2)校验agent与生成agent上下文隔离/独立执行，(3)'换架构而非堆门控'——一致性靠外部确定性校验(知识图谱/台账)，不靠AI自觉。
- **[high]** 对抗衰减的工程解法是'原子化拆分+确定性强制+独立验证+短信息链':把规则原子化降低单次加载量；用代码强制(但仅能验证外部行为)；用独立子agent做质量保障；信息传递链越短质量衰减越少(避免'产出→摘要→审核→摘要→编辑'的两次摘要损耗)。工作流编排(节点顺序/质检)与智能体架构(谁执行/上下文如何隔离)是两件事，不能用编排规则弥补架构缺陷。
  - 依据：同篇复盘的教训表:'规则不清→原子化拆分降低单次加载量(不是写更多规则)；执行不自觉→代码强制但仅限外部行为验证；自我认可(自欺)→换架构独立执行者(不是更多门控)'；'防退化机制的上限是架构约束而非机制数量'；'v1的信息链两次摘要损耗→v2合并相邻节点同上下文内部传递无损耗'。
  - 出处：https://cloud.tencent.com/developer/article/2650227
  - 启示：与用户CLAUDE.md的AI-pipeline铁律完全同构(LLM是被确定性代码包裹的子程序、硬约束由代码concat、独立验证)。pipeline应:生成与校验分属不同上下文的独立调用；一致性硬约束(设定/数值/伏笔)由代码注入和校验，不交给模型记忆。
- **[high]** 多agent长篇需要'正典事实分级治理'防AI幻觉:候选≠事实、推断≠事实、读者猜测≠事实；关键结果不能依赖未确认事实；正文canon通常高于台账；多agent交接必须分清'已确认/候选/待确认/冲突/废弃/推断'六态。配合'渐进式披露'(人物与设定按卷隐藏/开放)防止AI提前剧透幕后黑手/最终反转。
  - 依据：webnovel-handbook'正典事实、不确定性、设定冲突与AI幻觉控制'文档原文确认上述六态与'正文canon高于台账'；马良写作'渐进式披露:让AI写小说不再剧透的人物设定管理法(按卷隐藏与开放)'专文，解决'AI总把幕后黑手/隐藏金手指/最终反转提前写漏'。

### 最佳实践

- 顶层契约先行且不可变:让人类(或人机)先定'结局+核心冲突+主线锚点+力量/世界观底层规则'，存为canon顶层契约；所有章节生成只能在卷纲/章纲层填充，AI无权改写顶层锚点——这是防4-5章后崩的根。
- 用三级大纲schema做context分层:总纲→卷纲→章节蓝图(场景列表)。生成正文时只向AI喂'当前章节蓝图+知识图谱检索到的相关设定+前情压缩摘要'，不喂全书大纲；这同时解决长context导致的prompt泄漏和一致性衰减。
- 章节蓝图做成结构化字段(场景目的/冲突/关键行动/爽点类型与级别/钩子类型与内容/本章要回收或新埋的伏笔)，让AI'填表式'生成正文。约束住在字段里，正文只消费字段——从根上杜绝'把给AI的约束写进正文'。
- 场景切片化生成:一次只让AI写一个场景/分镜(如'雨夜敲陌生人的门')，而不是一次写完整章或吞几千字大纲。切片既提升细节质量、降AI味，又把单次加载量压到模型能稳定保持一致性的范围。
- 生成与校验上下文隔离:一致性/毒点/节奏校验必须由独立agent或独立调用执行，绝不让同一轮对话里'自写自审'(必然自欺)。校验结果以结构化findings回流，触发定点重写。
- 硬约束用确定性代码强制，不靠模型记忆:数值体系/力量等级/已确认设定/伏笔台账/平台节奏阈值/毒点黑名单，都由代码注入prompt并在产出后用规则扫描校验(参考用户CLAUDE.md的prompt+SUFFIX与control-signal模式)。
- 知识图谱给每个事实打'确认状态(已确认/候选/待确认/冲突/废弃/推断)'和'按卷可见性'两个标签;生成agent只能消费'已确认且当前卷可见'的事实，实现正典治理+渐进式披露(防AI提前剧透幕后黑手/最终反转)。
- 维护'情绪债账本'与'伏笔追踪表'两个跨章状态机:情绪债记录每条憋屈的强度/时长并强制对应释放强度≥债务(情绪守恒)；伏笔表记录出现章节/类型/回收计划/状态，临近回收章自动提醒。
- 平台化参数:把目标平台(番茄/起点/晋江/飞卢)做成开关，切换开篇节奏阈值、毒点容忍度、爽点密度刻度、章字数/日更节拍；用规则扫描器自动校验'第N字前是否出现冲突/金手指''每章是否≥1爽点''连续N章是否无情绪高峰'。
- 去AI味作为结构成立后的独立后处理pass:只改语言不改情节，针对句式重复/解释过满/段尾总结/同义反复/移动端段落无呼吸/旁白'万能讲解员'化做终磨;并跑AI含量自检以规避平台拒稿+追款风险。
- 数据反馈闭环:对接或模拟追读率/完读率，定位高跳出章节→自动触发该章重写;但把所有平台阈值当可调参数(平台算法不公开)，并保留'人类拍板是否重写/换书名简介(书测)'的决策点。
- 前情重提要'再激活'而非复述:旧信息通过当前压力/选择/物件/对话/后果重新带出，禁止把台账原样搬进正文、禁止用回顾吞掉付费章的新内容价值。

### 反模式

- 把几千字大纲一次性甩给AI让它续写整章/整书:它记不住上下文会开始'乱讲'，并把大纲里的设定/指令直接写进正文(正是用户遇到的prompt泄漏故障)。
- 让AI放手连续续写多章:自我一致性随篇幅必然衰减(伏笔忘、风格崩、字数膨胀)，4-5章后逻辑/人设/世界观崩——这是模型内在衰减，换更贵的模型也解决不了。
- 用'同一轮对话依次扮演策划/执笔/审核/编辑'冒充多agent:审核其实就是刚才的执笔者，不会真挑自己的错(自欺)，质量门形同虚设。
- 靠'写更多规则/堆更多门控'救架构缺陷:防退化机制的上限是架构约束而非机制数量，架构错了机制救不了；机制互相嵌套会让系统复杂度超过问题本身。
- 让AI决定剧情走向/节奏/情绪设计:实测AI给的是5-10年前被市场淘汰的旧套路，不懂在打斗里穿插人性博弈，言之无物;这些必须人类主导/拍板。
- 把爽点当玄学'写好人物'敷衍:爽点有可编码因果链(需求建立→压抑→释放)，忽略'情绪守恒'(憋屈久却只给短暂释放)、'爽点免费'(无铺垫直接开挂)、'虐点无补偿'都会劝退。
- 平均化节奏:AI默认匀速铺开(每段差不多长、每事件差不多详)，读起来寡淡;不在蓝图里显式标注加速/减速/急停，节奏必然'太平'。
- 忽略毒点黑名单:绿帽/送女/主角降智/圣母/视角混乱/注水任一即可让前面积攒的好感全废，甚至触发平台锁章/封号/下架(三观类)。
- 埋了伏笔忘回收 或 把伏笔当钩子用:前者=烂尾;后者=每埋一条都让角色大惊小怪，变成预告而非伏笔。无台账必翻车。
- 把未确认推断/候选/读者猜测当事实写进正文，或让设定库扁平无'确认状态/按卷可见性':前者制造AI幻觉式自相矛盾，后者导致提前剧透与前后设定冲突。
- 用起点思维写飞卢/用晋江风格套番茄:平台节奏阈值与毒点容忍度差异巨大(同一开头番茄扑街起点过签)，不参数化平台=系统性踩雷。
- 上架/付费节点不做专门校验:首免章不完整、首付费章卡断章拖答案/水化、或编造平台阈值，会制造'付费被骗感'直接掉订阅。

### 自动化判定

- **定核心冲突/结局/主线锚点/力量世界观底层规则(顶层契约canon)** = [human-only] — 一线作者毛志慧明确'让AI凭空创造打动人心剧情完全做不到'，剧情走向是人类核心价值；顶层一旦错，下游全崩，必须人类拍板。
- **题材/平台选择与市场判断** = [human-only] — 读者口味快速迭代(废柴退婚流已被嘲、圣母主角已被利己主角取代)，AI不掌握当下市场风向;可用AI做'领域匹配测试'辅助但最终人选。
- **分卷大纲/卷纲起草** = [semi-auto] — AI可在顶层契约约束下产出多版卷纲草案(蓝图Agent)，但节奏走向与卷高潮安排需人类拍板;AI单独做会节奏拖沓/套路化。
- **章节蓝图/场景级细纲(场景目的/冲突/行动/钩子字段)** = [semi-auto] — 结构化填表式最适合AI起草，但每章爽点/钩子/冲突升级由人类或人类设定的规则确认;蓝图是约束正文的契约层，错了会放大。
- **环境描写/旁白/场景氛围扩写** = [automate] — 作者一线共识'AI好用的就是环境描写/旁白/扩写细节';场景切片化下AI能产出丰富感官细节，质量稳定。
- **前情总结/再入场摘要/配角经历回顾** = [automate] — 毛志慧'AI承担总结性辅助(回顾前情/总结配角经历实力)'是明确的AI强项;但'再激活'式重提的写法需蓝图指定。
- **命名(功法/法宝/地名/人名/章节名候选)** = [automate] — 作者明确AI可'为武功法宝命名/生成名称';章节名/书名/简介可批量生成多版供人选。
- **正文章节生成(按蓝图场景切片)** = [semi-auto] — AI是核心写作引擎但需蓝图字段+知识图谱检索+前情摘要约束，且产出后必过一致性/毒点/节奏校验与人类抽检;放手连写必崩。
- **节奏/爽点密度设计与拍板** = [human-only] — AI默认匀速、不会主动制造高潮低谷;'压抑-释放'强度与时长的拿捏是人类主导项，AI只能按人标注的节奏指令执行。
- **打脸/装逼/情绪曲线/情绪债编排** = [semi-auto] — 机制可编码(需求→压抑→释放、情绪守恒)，AI可按编排填充;但'憋屈强度/释放时机'的设计与是否到位的判断需人类。
- **章末钩子生成** = [semi-auto] — 钩子有4种可模板化类型且应在蓝图预定，AI按'钩子字段+跨章问题链状态'生成最后三行很合适，人类抽检兑现度。
- **知识图谱/设定库实体抽取与维护** = [automate] — 马良知识图谱引擎已验证'自动提取角色/地点/物品/事件实体并建关系图';机械抽取适合自动，但'确认状态/按卷可见性'标签需人类或规则确认。
- **一致性校验(吃书/设定冲突/数值/时间线)** = [automate] — 独立一致性Agent+知识图谱实时检索是产品验证过的自动项;关键是必须独立上下文执行(非自审)并用确定性规则强制。
- **伏笔台账登记与回收提醒** = [automate] — 埋设自动登记、状态机流转、临近回收章提醒是纯机械任务;但'埋什么/何时回收'的伏笔设计是semi/human。
- **毒点/留存安全/三观红线扫描** = [automate] — 毒点是固定清单，可做规则扫描器+按平台切换容忍度，作为发布前硬gate;边界判断(虐点是否需补偿)可升级给人。
- **去AI味/句段终磨润色** = [semi-auto] — 独立后处理pass只改语言不改情节，AI可批量降AI味;但中文AI味重(GPT/Claude)，需人类终审+AI含量自检规避平台拒稿。
- **数据监控(追读率/完读率)→定位问题章** = [automate] — 指标采集与跳出章定位可自动化;但平台算法不公开，阈值需当可调参数。
- **据数据决定重写/换书名简介(书测)/是否上架** = [human-only] — 涉及商业判断与不可逆操作(上架、付费节点);追读率写崩仍可能平稳，需作者自己判断'有没有写崩'。
- **黄金三章/书名/简介打磨** = [semi-auto] — AI生成多版(番茄爆款公式)，但'最能落地的一版'由人选;开篇是签约生死线，节奏阈值需规则校验。

### 存疑

- GLM>GPT、DeepSeek'语言怪'的说法仅来自单一一线作者的实测，本流未找到独立可比的中文网文文笔横评(马良有'Claude/Gemini/GPT/DeepSeek实测排名'文但未取到具体打分);需要一份控制变量的盲测(同蓝图同题材)来定模型选型，不能照搬营销话术。
- 追读率1.8章/4.7章一个情绪高峰、番茄三章追读率35/45%等阈值多来自单一来源或内部流出资料，平台算法不公开;需要把这些当可调参数并用自有数据校准，'真值'无法外部确认。
- '设定集/知识图谱'对生成质量的真实增益缺乏量化对照:有了知识图谱后吃书率下降多少、对追读率有无影响，现有材料只有用户证言(testimonial)，没有A/B数据。
- 女频(晋江情感线/CP张力/人物独立性)与男频(爽点/升级/打脸)的pipeline约束差异很大，本流证据偏男频;女频的'情绪共鸣/感情拉扯'是否同样可编码、毒点清单是否不同，需要专门补研。
- 'AI含量检测'的具体判定方法与阈值(<20%不计/>40%拒稿)各平台不一且在变，且检测器对'AI起草+人工深度改写'的判定边界不清;pipeline的去AI味要做到什么程度才安全，需要持续跟踪平台政策。
- 原创性/防洗稿边界:pipeline联网拆竞品/吸收风格样例时，'学习类型语法/读者承诺'与'抄具体表达/桥段组合/专名体系'的红线如何在工程上自动判定，webnovel-handbook给了原则但缺可执行的检测手段。

## 调研流：网文分品类的结构与套路库（品类感知的约束模板/检查清单）

**摘要**：不同品类网文在"爽点引擎、开篇必交代项、结构骨架(单元剧/长线/升级阶梯)、最易写崩点"四个维度差异极大，pipeline 必须"品类感知"：第 0 步由人类锁定品类+子类+调性，系统据此挂载一整套约束模板（爽点引擎单句锚、力量/数值 schema、开篇平台参数表、毒点黑名单）。开篇"黄金三章"与"节奏=给读者奖励的频率"是跨品类硬约束，但参数随平台变化（番茄前300字出冲突/前1000字亮金手指、起点可慢热但第三章前必亮金手指、飞卢前300字必须有炸点），应参数化而非写死。结构骨架可二分：升级阶梯（玄幻/都市升级，换地图+三段大情节）与单元剧（无限流/规则怪谈/悬疑/电竞/经营诸天，副本/案/赛为骨+长线暗线），pipeline 应按骨架类型切换大纲生成器。AI 在每个品类的崩点高度可预测且大多是"硬一致性"问题（战力通胀/境界倒退、隐藏身份/叙诡真相泄漏、经济体系崩、副本规则不自洽），可用显式 schema+知识图谱+毒点 lint 自动兜底；而剧情走向、节奏曲线、爽点拍板是 AI 最弱项，须人类主导——印证一线作者实践（环境/旁白/扩写交 AI，走向/节奏交人）。


### 关键发现

- **[high]** 玄幻/仙侠/修真(升级流)的爽点引擎是'升级打怪+装逼打脸+降维碾压'，结构骨架是'升级阶梯换地图+全书拉3段大情节(第一段30万字前解决)'；AI 最易崩在战力/境界体系通胀与'吃书'(配角上一章筑基下一章练气)。
  - 依据：起点作家专区《玄幻小说创作入门指南》把主角分废材崛起/土著成长/强者重生/夺舍废材四型，均为'由弱到强逐级升级、抑后扬';起点社区2025武侠仙侠盘点反复出现'升级速度一压再压''到宗门后开始套娃循环往复水漫金山';腾讯云AI写小说实操贴明确要求'反派前一章筑基下一章练气，爽感就崩塌'。
  - 出处：https://write.qq.com/portal/content/5337117403202901 ; https://www.qidianclub.com/threads/1882 ; https://cloud.tencent.com/developer/article/2626811
  - 启示：玄幻品类模板必须强制一个显式'境界表+全书最高战力上限'schema，并由代码在每章生成后校验单调性；大纲生成器按'三段大情节(30万/中盘/收尾)+换地图'节点产出，而非线性堆章。
- **[high]** 都市(赘婿/战神/神医/兵王)的爽点引擎是'隐藏身份揭露+打脸四部曲(挑衅→隐忍→反转→余震)+扮猪吃老虎'，骨架是单元剧式打脸循环(番茄5-8章/起点15-20章/飞卢3章一个周期)；AI 最易崩在'马甲(隐藏身份)过早泄漏'毁掉反转，以及打脸节奏被写得太均匀。
  - 依据：wangwen666《脑洞爽文5大技法》给出'打脸四部曲'结构与各平台周期参数;知乎黄宇成答主指出战神/赘婿爽点集中在'身份揭露、系统、扮猪吃虎'且'爽点越密越易追读';中国作家网微短剧文记载赘婿/战神爆款均以'扮猪吃老虎/打脸复仇'+末尾揭露主角隐藏身份(帝师/股神)为核心手法。
  - 出处：https://wangwen666.com/post/186.html ; https://www.zhihu.com/question/430082486/answer/3573227906 ; https://www.chinawriter.com.cn/n1/2024/0806/c404027-40293380.html
  - 启示：都市品类模板需把'隐藏身份/真实实力'登记为对 AI 隐藏的可见性条目(防泄漏)，把打脸单元做成'挑衅-隐忍-反转-余震'四段卡模板，并把'每周期章数'设为平台参数；节奏由人类在章纲标注，避免 AI 把每次打脸写成等强度。
- **[high]** 系统流的本质是'金手指显形化+不讲道理+网游式升级感'，金手指从隐形走到叙事起点、成为'规则之外但有稳定预期的规则'；AI 最易崩在让系统规则随意漂移、或用大量系统对话拖慢节奏。
  - 依据：起点社区《核心要点——系统流》指出系统流核心是'升级感'与'不讲道理'(花一万得十万也没毛病)，但优势在'有稳定预期';中国作家网《'爽'及其完成》分析金手指已从'规则之外的意外'变为'规则之外的规则'，一开始就有明确运作规则;起点作家专区签到流写法警告'不要让主角与系统大量不必要对话拖慢节奏'。
  - 出处：https://www.qidianclub.com/threads/125 ; https://www.chinawriter.com.cn/n1/2024/1128/c404027-40370888.html ; https://write.qq.com/portal/content/5337117403202901
  - 启示：系统流模板应把'系统规则'写成一份不可变的契约文件(奖励公式、触发条件、限制)注入每章并代码校验，禁止 AI 临场改规则；系统播报做成固定短模板，防注水对话。
- **[high]** 无限流/规则怪谈是单元剧(副本制)骨架，每副本三段式(起因/经过/结果=爽点预热/爆发/回味)，爽点引擎是'主角解读并利用规则破局+恐怖氛围+解谜反转'；AI 最易崩在'每个副本规则的内部自洽'、副本与主线割裂、以及恐怖氛围被爽点稀释。
  - 依据：lkong猎空《如何打造吸睛副本》给出副本三段式与'每个副本结束都会掉追读，要提前拉期待留钩子';规则怪谈对比文强调'规则怪谈核心魅力恰恰是规则的不可靠，主角在矛盾规则中求生';北京新京报/北京新闻记《十日终焉》报道指'规则怪谈+无限流'将规则由类型变元素、文本'数据库化'。
  - 出处：https://www.lkong.org/thread-5-1-1.html ; https://write-on-paper.wikidot.com/lun ; https://m.bjnews.com.cn/detail/1735271251168423.html
  - 启示：无限流模板需为每个副本生成独立的'规则清单+漏洞(破局点)'结构化卡，并对该副本内一致性单独校验；主线暗线与副本明线分两条记忆线管理；可把副本做成可组合的'事件卡数据库'供人工挑选，单元化天然降低跨章一致性负担。
- **[high]** 科幻/末世(丧尸/废土/星际)的爽点引擎是'艰难求生+基建or丛林法则+进化升级'，进化体系即升级阶梯且'最高战力必须开局锁定一本写完不超'；经典毒点是父母双全、后宫开太多、敌人不死光环、政府无差别崩溃。
  - 依据：搜狐《末世丧尸文常用设定》逐条列出毒点:父母双全限制主角行动、后宫开太多导致剧情乱套、进化体系不锁最高战力必崩、敌人不死光环、政权无理由崩溃;台北大学末世论文与起点书单引读者吐槽'主角金手指大开大杀四方就不叫末世，艰难求生才是末世底色'。
  - 出处：https://www.sohu.com/a/512862136_120099877 ; https://www.cl.ntpu.edu.tw/ntpu-20/j-2803.pdf
  - 启示：末世模板把'最高进化等级/战力上限'作为开局锁定常量校验(同升级流)，把'父母/后宫/敌人复活/政府状态'写进品类毒点黑名单做自动 lint；调性(艰难求生 vs 爽囤货)由人类选定后影响整本资源稀缺度参数。
- **[high]** 悬疑/推理的爽点引擎是'揭秘/反转的那一刻+叙述性诡计的信息差'，骨架常为单元案+长线主谜；AI 最易崩在叙诡需要作者对读者全局性'刻意隐瞒/误导'——AI 不知道哪些信息该藏，极易提前泄底或埋错公平线索。
  - 依据：百度百科/机核/萌娘百科叙诡词条一致定义:叙诡是'作者利用文章结构或文字技巧对读者隐瞒/误导关键信息'，分点(人物/时间/地点)与面(结构)两类，'点一句话就能泄底';新京报文强调'真相揭晓前作者完全站在凶手一方'、需精准控制何时揭露。

### 最佳实践

- 把'品类选择'设为 pipeline 第0步且 human-only：人类锁定品类→子类(玄幻-洪荒/凡人/签到/苟道；都市-赘婿/战神/系统/神医；女频-古言/现言/宅斗/穿越；末世-丧尸/废土/星际…)→调性，系统据此挂载整套约束模板。
- 为每本书生成一句'爽点引擎'锚并注入每章生成:玄幻=升级碾压打脸；都市=隐藏身份揭露+打脸四部曲；系统流=不讲道理的升级感；无限流/规则怪谈=规则破局+解谜反转；悬疑=反转揭秘；末世=艰难求生+进化；种田/经营=获得感+资源积累；女频=情感拉扯/虐渣+大女主事业；电竞=策略碾压+团队荣誉。
- 开篇做成'平台×品类'参数化模板:可配置 主角出场字数上限、核心冲突时限、金手指亮相时限、章末钩子类型(悬念/冲突升级/爽点预告)、开篇法(先抑后扬/先扬后抑/复仇/烘托/楔子)；自动产出多版本供人选。
- 升级类品类(玄幻/系统/末世/电竞)强制一份显式数值 schema(境界表、全书最高战力上限、装备/系统数值)，开局锁定上限并由代码在每章生成后校验单调性，杜绝战力通胀与境界倒退。
- 用'可见性分级/真相金库'统一管理隐藏信息:都市赘婿的隐藏身份、悬疑的叙诡真相与凶手、女频重生/穿越者的已知未来、长线伏笔，全部标记为'对生成AI隐藏、仅向大纲层开放、到某卷才解锁'，并配伏笔回收台账。
- 把重复性单元用'三段式工业卡'生成:打脸=挑衅→隐忍→反转→余震；副本=爽点预热→爆发→回味(起因/经过/结果)；推理章=线索A→推翻→更大秘密；用倒推法从爽点高潮反推单元大纲。
- 节奏交人/蓝图不交AI:在章纲显式标注'本章加速/减速''本章爽点=X''章末钩子=Y'，并把'每章≥1小爽点+必留钩子、每3-5章1大爽点'做成生成后自动校验，针对AI'节奏太均匀'的缺陷。
- 按骨架类型切换大纲生成器:升级阶梯(换地图+全书3段大情节,第一段30万字前解决)用于玄幻/都市升级；单元剧(副本/案/赛/客为骨+长线暗线)用于无限流/规则怪谈/悬疑/电竞/经营诸天；慢热长线(经济账本)用于种田/经营/古言。
- 慢热品类(种田/经营/古言)允许铺垫但禁止注水:维护'经济账本'结构并设硬约束'收获永远跟不上消耗、金手指必须有限制'，且每章至少1个获得感锚点。
- 人机分工底线(印证一线作者实践):环境描写/旁白/细节扩写交AI自动化；剧情走向、节奏曲线、爽点位置、关键反转/对局/感情推拉由人类拍板；模型按'品类×平台×环节'评测矩阵选(慢热细腻环节可用DeepSeek，快节奏品类要被节奏约束强驱)。
- 为每个品类配'毒点黑名单'做自动lint:如末世(父母双全/后宫过多/敌人不死光环/政府无理由崩)、玄幻(战力崩/配角脸谱化)、女频(男主前期戏份失衡/配角降智)、通用(prompt泄漏到正文/注水/生僻AI味词)。
- 把约束/context 与正文严格分离以防prompt泄漏:设定/规则/可见性走结构化注入(知识图谱+RAG按需装配)而非把长设定塞进章节prompt正文，输出层只产正文，杜绝AI把约束当正文写出来。

### 反模式

- 让AI自由连续续写4-5章后必崩(逻辑/人设/世界观)，根因=上下文窗口装不下全书+无外部记忆锚点+风格无约束；解法是外部记忆(知识图谱)+按需装配，而非一味换更大窗口模型。
- 章节prompt过长→AI把本应作约束/context的设定、规则、世界观当正文写进章节(prompt泄漏)，尤其在把大段设定一股脑塞进单次prompt时。
- 升级类品类不锁'全书最高战力/境界上限'→战力通胀、境界倒退、配角等级前后矛盾(筑基→练气)，爽感直接崩塌。
- 隐藏身份(赘婿马甲)、叙诡真相、长线伏笔被AI过早泄漏(AI不知道哪些该藏)→反转/打脸/解谜失效，整本废掉。
- 用一个品类/平台的节奏套另一个(飞卢卡高潮的节奏拿到起点会一泻千里、起点慢热节奏发番茄前10章还在铺设定就被划走)；AI默认节奏'太均匀'不会主动加减速。
- 开篇铺垫太多:穿越前写三五万字废笔再'biu'一声穿越全丢、开篇堆世界观/多人物名(三章内>5个名字)、大段景物描写秀文笔→读者30秒内划走、编辑拒签。
- 末世经典毒点:主角父母双全(限制行动自由)、后宫开太多(剧情乱套、女角十项全能)、敌人开不死光环假死再登场、政府无差别崩溃。
- 种田/经营经济体系崩:金手指开太大、收获超过需求→期待感消失(收获必须永远跟不上消耗)。
- 配角全脸谱化/NPC化(只为主角打脸服务、忽略其复杂心理)→更高层次的'老白'读者不买账，且与'网文向更高层次跃升、复调群像'趋势相悖。
- 把'换个更大上下文模型'当作一致性银弹→窗口再大也总有上限且既贵又慢，仍会吃书；正解是分类上下文构建+RAG按需装配。
- 迷信单一模型全局评价(GLM>GPT/DeepSeek语言怪)→实为'品类×平台×环节'相关:DeepSeek慢热细腻可能适合起点描写环节却不适合番茄快节奏；应做评测矩阵而非一刀切。
- 把节奏/剧情走向/爽点拍板交给AI自动决定→这是AI最弱、最易翻车的部分，必须人类主导(与环境/旁白/扩写可交AI形成对比)。

### 自动化判定

- **选定品类+子类+调性(玄幻-洪荒/凡人/苟道；都市-赘婿/战神/系统；女频-古言/宅斗；末世-丧尸/废土…)** = [human-only] — 品类选择决定整套约束模板、爽点引擎、目标平台与市场定位，且一线实践证明剧情走向/定调是AI最不可靠处，须人类拍板。
- **生成开篇'黄金三章'骨架(按 平台×品类 参数表，多版本)** = [automate] — 开篇高度套路化、参数明确(主角出场/冲突/金手指时限/钩子类型)，可全自动产出多版本供人选，是ROI最高的自动化环节。
- **生成品类力量/数值体系 schema(境界表/战力上限/系统数值/装备段位)** = [semi-auto] — AI可按品类模板草拟，但'全书最高上限'等关键常量须人类锁定，之后交给代码校验单调性。
- **生成单元卡(副本/案件/打脸/赛事)的三段式骨架** = [automate] — 三段式(预热/爆发/回味、挑衅/隐忍/反转/余震)是可复用工业模板，单元化天然降低跨章一致性负担，可批量自动生成入库供挑。
- **把单元串成成长线、拍板节奏曲线与爽点/反转位置** = [human-only] — 节奏把控与剧情走向是AI最弱项(默认太均匀)，且关键反转/对局/感情推拉决定成败，须人类主导，仅由AI在约束内填充。
- **环境描写、旁白、细节扩写、过渡段填充** = [automate] — 一线作者实践已证明AI在此处好用，属可放手自动化的低风险高产出环节(慢热细腻品类可优先用描写强的模型)。
- **隐藏信息/伏笔的可见性分级与剧透过滤** = [semi-auto] — '哪些条目对AI隐藏、属于哪卷开放'的规则可代码化执行，但'何时揭露/如何揭露'(尤其叙诡、马甲反转)由人类设定。
- **一致性校验(吃书/战力崩/名字/角色状态/时间线/伏笔回收)** = [automate] — 可用知识图谱+RAG检索+数值规则+冲突预警自动跑，是机器最擅长、最该承担的兜底环节。
- **品类毒点扫描(按黑名单lint)+AI味/注水/生僻词质检** = [automate] — 毒点黑名单与文风偏差可对照爆款样本做成自动检查项，生成后即时报警，无需人工逐条排查。
- **章节正文生成(骨架内填充)** = [semi-auto] — AI在人类约束的骨架/节奏/可见性内批量产出正文，但高潮章、关键转折、感情高软度段落须人类把关与重写。
- **模型/平台选择(GLM/DeepSeek/GPT/Claude × 番茄/起点/晋江 × 描写/节奏/对话环节)** = [semi-auto] — 应由'品类×平台×环节'评测矩阵驱动而非全局结论；矩阵可自动跑评测，但最终路由策略由人类按调性确认。

## 调研流：AI长篇/长上下文故事生成的学术与工程方法

**摘要**：学术界已经把"让AI写长篇还不崩"收敛成一套稳定范式：先用分层大纲规划（Re3/DOC/CONCOCT），再用"重提示+滚动记忆"逐段生成（RecurrentGPT），生成后用独立的一致性检查层做矛盾检测（ConStory-Checker/FACTTRACK），记忆靠递归摘要+RAG检索压缩——这与该作者"每5章压缩context+更新memory.md+开新window"的土法实践几乎一一对应，说明其经验路线是对的，只是缺工程化骨架。最关键的设计约束有三：(1) AI永远是被确定性代码包裹的子程序，剧情走向/节奏/爽点由人或大纲层拍板，AI只负责被大纲约束的段落扩写——这与"剧情AI不好用、描写AI好用"的一线判断完全吻合；(2) prompt泄漏到正文的根因是LLM架构上没有指令/数据的形式化分隔（ICLR 2025已证），章节级长prompt必须拆成 system约束 + 检索context + 滚动摘要 + 只输出正文的硬后缀，而非一坨；(3) 一致性错误不是均匀分布——实测集中在"事实+时间线"维度、出现在叙事中段(40-60%)和高熵生成处，校验资源应定向投放。"AI味"是中文生成的模板化句式特征（连接词堆砌/对仗/总结收束/光明的尾巴），用同源大模型互洗无效，需要Show-Don't-Tell式推理前置+感官/方言注入+人工精修金句钩子。


### 关键发现

- **[high]** 长篇生成的可靠范式是'分层规划→重提示逐段生成→修订'四段式，而非让模型一次性长输出或盲目续写；动态把'大纲+当前故事状态'重新注入每段prompt是核心机制。
  - 依据：Re3(EMNLP 2022)把人类写作分解为Plan-Draft-Rewrite-Edit四模块：先让通用LLM构造结构化大纲(premise→setting→characters→plot points)，再'反复把大纲与当前故事状态的上下文注入prompt'生成段落，对多个续写候选按情节连贯性/前提相关性重排序，最后编辑修正事实一致性。相比同长度直接生成，人评连贯性+14%、相关性+20%，生成2000+词。DOC(ACL 2023)在此之上加'详细大纲器+基于FUDGE的细粒度控制器(longformer分类器做reranker/controller)'，平均3500+词，连贯/相关/有趣度都显著超Re3。
  - 出处：https://arxiv.org/abs/2210.06774 ; https://aclanthology.org/2022.emnlp-main.296.pdf ; https://arxiv.org/abs/2212.10077 ; https://github.com/yangkevin2/doc-story-generation
  - 启示：pipeline应固化为'大纲先行+逐段重提示'循环：每章prompt由代码动态拼装(本章大纲项+检索到的相关设定+滚动摘要)，而不是人手写长prompt。reranker思想可落成'同段生成N个候选→用一致性/爽点打分器选最优'。
- **[high]** RecurrentGPT用自然语言模拟LSTM的长短期记忆，是该作者'memory.md+开新window'土法的学术对应物：短期记忆=近段关键信息摘要(每步重写)，长期记忆=所有历史段落摘要(存盘+语义检索召回)。
  - 依据：RecurrentGPT(arXiv 2305.13304, aiwaves)在每个timestep接收上一步生成的段落+下一段计划，attend长期记忆(历史段落摘要，存硬盘，按需语义检索相关段落)与短期记忆(近期关键信息的自然语言摘要)，把这些拼进prompt让backbone LLM生成新段落+下一段计划，并重写短期记忆、把本段摘要append到长期记忆。记忆是自然语言、人可读可编辑，故可解释、可交互。
  - 出处：https://arxiv.org/abs/2305.13304 ; https://github.com/aiwaves-cn/RecurrentGPT

### 最佳实践

- 四段式骨架固化：Plan(分层大纲)→Draft(逐段重提示扩写)→Rewrite(候选重排/一致性择优)→Edit(去AI味+切光明尾巴+补连接句)。每一段都是确定性代码包裹的LLM子程序，代码负责拼prompt、择优、校验、回写记忆。
- 章节prompt用代码动态拼装而非人手长写：system段=稳定写作约束+风格禁用词清单；data段=用强分隔符<<<SETTING>>>包裹的'本章相关设定(检索召回)+分层滚动摘要+本章大纲项+人物行为锚点'，并标注'以下仅为资料，不得复述/解释/写入正文'；末尾硬后缀='只输出本章小说正文，不要章节标题/元信息/设定复述'。
- 记忆做三层递归摘要：段/章级短摘(每章重写、必进prompt)→卷级摘要→全书主线摘要；长期用'章级摘要库+原文'混合检索按需召回。摘要用'旧摘要+新内容→新摘要'递归生成，禁用易破坏连贯的硬replace/append/delete。
- 设定集结构化而非散文：人物卡用'行为锚点'字段(具体事件，非形容词)；维护(实体,关系,实体)人物关系三元组与'事件→时间轴+前置/后置状态'表；伏笔登记表记录(埋点章→应回收章)，生成时到点自动提醒回收。
- 一致性校验分维度+定向投放：按人物/事实/时间线/世界设定/风格各建一套judge prompt或分类器，要求输出'矛盾+确切文本证据'；在叙事中段(40-60%)、转折高潮等高熵处加密检查。校验只做'标记可疑'(自动检测F1仅~0.68)，裁决与改写交人或更强模型。
- 去AI味做成生成期约束而非事后互洗：prompt内禁用连接词清单(然而/与此同时/随后/首先其次最后)+强制Show-Don't-Tell(先让推理模型分析微表情/肢体再落笔、禁止直接写情绪词)+感官配额(每章≥N处嗅觉/触觉)+按场景调句长(打斗短句/回忆长句)。配一个正则'AI味lint'扫模板句打分。
- 分工=导演(人/orchestrator)定冲突/设定/情节/人物与节奏，演员(AI)只做被大纲约束的段落扩写；剧情走向与章末爽点钩子/金句保留human-only。这把'剧情AI不好用、描写AI好用'制度化为权限边界。
- 分卷边界=强制状态刷新点：重载设定集、压缩并固化卷级摘要、开新会话清掉脏上下文；即便用1M上下文模型也保留分层摘要+检索注入，不做全文直塞。
- 模型可注入+本地A/B选型：client/base_url/model做成可换；对关键段用GLM/DeepSeek/Claude各跑一遍，以'AI味lint分+一致性judge+人评钩子强度'三指标实测选，不信单一经验或厂商跑分。
- 同段重试上限设低(避免6-7次后自我重复塌缩)，改进靠'换prompt/换模型/改写候选'而非反复重抽同一prompt。

### 反模式

- 盲目连续续写(rolling baseline)：不注入大纲/状态、只把上一章尾巴喂进去续，Re3论文已证其连贯性显著劣于重提示——这正是'4-5章后世界观全崩'的直接成因。
- 把所有设定/约束塞进一个超长章节prompt：触发LLM的指令/数据无分隔缺陷，约束被当指令写进正文(prompt泄漏)；越长越泄漏，且挤占生成预算。
- 用同一个/同类通用大模型给自己生成的文本'去AI味'：实测改写后AI检测率仍95-98%，只是把AI味A换成AI味B，纯属浪费一轮调用。
- 全文直塞百万字长上下文当作免一致性工程：中段和高熵处照崩、成本二次增长、模型难聚焦；长上下文不能替代分层摘要+检索+状态追踪。
- 只在章末或全文做一次性一致性扫：错误集中在中段且分维度差异大(叙事连贯检测F1仅0.51)，均匀/末端单遍扫会漏掉中段矛盾与时间线冲突。
- 用形容词定义人物('很勇敢/很腹黑')：模型无锚点可依，行为漂移；必须换成可被检索和自检的具体行为锚点。
- 过度工程化的多Agent堆砌(novelix式overengineered)：Agent众多但控制权/数据流不清、人拍板的接口缺失，结果既不可控也不好用——分工要少而清，orchestrator与人的拍板点必须显式。
- 让AI包办节奏与章末钩子：AI默认匀速叙事+给每段加'光明的尾巴'(升华收束)，爽点写不够狠；这些是human-only或需人精修，全自动会出戏。
- 信任厂商创作类自评/跑分做选型：去AI味横评、写作文笔排名多为自利数据或编码基准外推，必须以本地A/B+人评落地。
- 把自动一致性检测的输出当终判直接自动改写：F1~0.68意味着约1/3误差，自动'修'可能引入新矛盾；应人/强模型裁决。

### 自动化判定

- **分层大纲/卷骨架/拆章(剧情走向)** = [semi-auto] — AI可生成多套分层大纲候选，但剧情走向/主线必须人拍板——与一线'整体结构由人决定'一致；可用具体度评估器辅助标记节奏失衡处。
- **章级prompt上下文拼装(检索召回相关设定+滚动摘要+大纲项+硬后缀)** = [automate] — 纯确定性代码：检索、模板填充、分隔符包裹、后缀拼接、JSON包装剥离，无需人介入，正是防prompt泄漏的工程层。
- **环境描写/旁白/细节扩写(被大纲约束的段落正文)** = [automate] — 一线明确AI擅长此项(王峰'细节描写由AI完成')；只要被大纲与设定约束、生成后过校验即可自动。
- **递归摘要/分层记忆压缩与回写** = [automate] — M_i=LLM(H_i,M_{i-1})是确定性可包裹的循环，鲁棒且与检索/长上下文正交；人只需偶尔审阅可读摘要。
- **设定集/时间线/人物状态抽取与更新** = [semi-auto] — AI从新生成章节抽取事件/状态回写时间轴与三元组，但抽取F1有限、易错，关键设定变更需人确认。
- **分维度一致性检测(人物/事实/时间/世界/风格)** = [semi-auto] — 自动检测+证据落地可做'标记可疑'，但整体F1~0.68，最终裁决与改写需人或更强模型，不能全自动改。
- **RAG检索注入(向量+知识图谱混合召回相关设定/人物关系)** = [automate] — 检索与注入是确定性流程；图谱召回处理同名消歧与跨章关系，全程无需人。
- **去AI味润色(模板句lint+Show-Don't-Tell改写+切光明尾巴)** = [semi-auto] — 句式层面的去模板化可自动(lint+改写pass)，但金句/特有黑话/情感峰值仍需人精修，同源大模型互洗无效。
- **节奏/篇幅预算分配** = [semi-auto] — CONCOCT式具体度均衡可自动给建议(标记一笔带过的重要事件/过度展开的琐碎)，但最终节奏由人定——一线判断节奏需人主导。
- **章末爽点/钩子/悬念/金句** = [human-only] — AI默认匀速且写不够狠、爱加升华收束；爽感强度是网文命门，必须人打磨或重度精修。
- **输出格式约束执行(只输出正文、剥离元信息)** = [automate] — 代码硬拼后缀+结构化输出字段提取+正则剥离设定复述，确定性强，是防泄漏的最后一道闸。
- **模型选型A/B(GLM/DeepSeek/Claude)** = [semi-auto] — 自动跑三指标(AI味lint/一致性judge/钩子强度)给候选排序，但最终采用看人评，厂商榜不可信。

### 可复用产物

- Re3 Plan-Draft-Rewrite-Edit四模块 + 'reranker按情节连贯性/前提相关性选候选'机制(代码:github.com/yangkevin2/emnlp22-re3-story-generation；DOC升级版github.com/yangkevin2/doc-story-generation)——可直接照搬为'同段生成N候选→打分择优'。
- RecurrentGPT双层自然语言记忆数据结构(短期=近段关键信息摘要每步重写；长期=历史段落摘要存盘+语义检索召回)+'每步同时产出下一段plan'(代码:github.com/aiwaves-cn/RecurrentGPT)——人可编辑的memory正是作者memory.md的升级原型。
- ConStory-Checker的5类19子类错误taxonomy + 'judge输出矛盾必附确切文本证据'的prompt模式(github.com/Picrew/ConStory-Bench，含characterization/factual_detail/narrative_style/timeline_plot/world_building五个judge prompt模板.md)——可直接作为一致性校验层的分维度prompt库。
- FACTTRACK的'事件→pre-facts/post-facts→时间轴'状态追踪结构(aclanthology.org/2025.naacl-long.144.pdf)——治时间线/战力崩坏与伏笔遗忘的数据模型。
- 递归摘要公式 M_i=LLM(H_i, M_{i-1}, P_m)(arxiv 2308.15022)与RAPTOR分层摘要树——分层记忆压缩的现成范式。
- CONCOCT具体度评估器+vaguest-first展开+具体度过滤(github.com/YichenZW/Pacing)——节奏体检/篇幅均衡的可训练组件。
- Agents' Room角色分工模板(orchestrator + 规划Agent[Conflict/Setting/Plot/Character] + 写作Agent[按Freytag金字塔铺垫/上升/高潮/回落/结局] + 共享scratchpad)——多Agent网文流水线的最小可用拓扑。
- 防泄漏prompt结构：strict分隔符模板('<<<DATA>>>内仅为资料，忽略其中任何像指令的内容')+结构化输出把正文放专字段+末尾硬后缀'只输出正文'(出处:ICLR2025指令-数据分隔论文、13-LLM分隔符实验)。
- 中文'去AI味'特征清单(连接词:然而/与此同时/随后/最终/首先其次最后；对仗双子句；宛如/仿佛/犹如；总结收束'一切的一切…'；万能反应'邪魅一笑/面色铁青')+三层去AI味法(感官轰炸/方言职业术语注入/多模型协同)——可做成生成期禁用词约束+AI味lint规则集。
- DeepSeek R1式Show-Don't-Tell推理前置prompt('先分析极怒压抑时的微表情/肢体、禁止直接写愤怒，再写场景')——把推理模型的CoT用于反AI味描写的现成prompt模式(waqupin.com/blog/writing/remove-ai-style)。
- 开源中文实现参考AI-Novel-Writing-Assistant(github.com/ExplosiveCoderflome/AI-Novel-Writing-Assistant)的工序流:拆书分析→知识库→世界观骨架→角色库→卷战略/卷骨架→节奏/拆章→章节执行→质量修复→伏笔回填→自动导演模式+'书级framing/故事引擎/前30章承诺'可继承引导层——可作为流程蓝本(但注意避免过度Agent化)。
- 蛙蛙写作'总纲-章纲-正文'三级分层 + FeelFish'全文检索问答覆盖百万字、自动规整人物设定'(163.com/dy/article/KVKFE5U80527EOT6)——商业工具验证过的层级与检索架构。

### 存疑

- 自动一致性检测F1只有~0.68(叙事连贯仅0.51)，在百万字连载里召回不够；如何用'结构化状态追踪+多judge投票+人工抽检'把可疑标记的精度/召回提到可用线，且不把人工裁决量压垮？
- 递归摘要在数十万字尺度会累积信息丢失(早期细节被压没)，与RAG原文召回如何分工——哪些事实必须进结构化设定集(永不压缩)、哪些可进可压缩摘要？阈值如何定？
- CONCOCT能均衡大纲节奏，但网文要的是'爽点节奏'(蓄势-爆发-钩子)而非均匀节奏；如何把网文特有的章末钩子/打脸节拍编码成可控制的pacing目标，而不是追求uniform pacing？
- 'GLM中文文笔>GPT'缺创作专项公开评测，且模型迭代快(GLM-5.2/DeepSeek-V4)；需要一套可复用的中文网文创作评测(AI味/一致性/爽感/术语契合)来持续做选型，而非依赖经验或编码基准。
- 厂商去AI味横评(马良80% vs DeepSeek 97%)为自利数据；专用'去AI味'微调模型相对'生成期约束'的真实增量有多大，是否值得引入额外一跳？需独立复现。
- 知识图谱式人物关系/时间线的构建与动态维护成本高(每章抽取+消歧+回写)，在日更万字的网文节奏下能否实时跟上？轻量化(只追踪核心实体/关键状态)到什么粒度仍有效？
- '光明的尾巴'、论文腔等是RLHF对齐的副产物还是可纯prompt根除？若是对齐倾向，是否需要局部微调或负样本约束才能稳定切除，纯后处理切尾会不会误伤合理收束？
- 人机协作的最优介入密度未知：王峰是2000prompt+全程人工润色，作者是每5章一次重度介入；在质量/产能曲线上，'每章轻介入'vs'每卷重介入'哪种对长期一致性与爽感更优，缺对照实验。

## 调研流：现有AI写小说工具/项目的批判性拆解

**摘要**：拆解后最关键的结论：市面所有工具（Sudowrite/NovelCrafter/NovelAI/AI Dungeon）本质都把"一致性"当成"检索问题"——把设定存起来、按需喂给模型、然后信任模型自己用对——这是长篇必崩的根因，因为存储+注入≠强制执行（active enforcement）。novelix 的过度工程恰恰是因为它想做"强制执行"却用了10个agent+7个truth文件+33维审计+22条改写规则，复杂度爆炸、脆弱难用；而真正可借鉴的复杂度边界是：外部结构化记忆（知识图谱/canon.md）+ 按场景分维度装配上下文（完整/仅名称+token配额）+ 关键词/场景触发注入 + 生成后对照canon做轻量审计，这套组合用一个有状态的确定性循环就能实现，不需要十个agent。prompt泄漏到正文的根因是LLM架构层面"指令与数据不分"，章节prompt越长、指令越多越容易把约束写进正文（《天命使徒》团队亲述"提示词长了不行短了也不行"，已发表AI网文里残留指令/符号正是平台一秒鉴AI的特征）。分工边界被一线实践反复验证：人定结构/剧情走向/节奏/防剧透，AI填环境描写/细节扩写/旁白（《天命使徒》AI占70%但"整体结构由人决定，细节描写由AI完成"）。对抗过度工程的正解是：把"不该交给运气的事"（记忆、装配、剧透红线、文风Persona）交给系统，把灵感和节奏留给人，而不是追求一键全自动。


### 关键发现

- **[high]** novelix的过度工程具体体现在：10个串行agent(Radar/Planner/Composer/Architect/Writer/Observer/Reflector/Normalizer/Auditor/Reviser)+7个Markdown truth文件+可选SQLite+33维连续性审计+22条反AI检测改写规则，仅状态管理就有Observer→Reflector→Normalizer→Auditor四级抽取-校验流水线。正确的复杂度边界应是：把10个agent合并成3-4个角色（写作/抽取/审计/修订），truth文件保留为canon数据结构而非agent链
  - 依据：WebFetch该repo README：'10-agent sequential pipeline','7 Markdown truth files(current_state/particle_ledger/pending_hooks/chapter_summaries/subplot_board/emotional_arcs/character_matrix)','33-dimensional audit','22 rewrite rules';过度工程信号显式列出'Agent Proliferation: 10 sequential agents where many could combine(Writer+Normalizer, Planner+Composer)'、'4-stage extraction-validation pipeline just for state management'
  - 出处：https://github.com/zxerai/novelix
  - 启示：pipeline设计应抄它的canon数据模型（7类truth的taxonomy很好用），但坚决拒绝它的agent数量。复杂度预算放在'状态结构'而非'agent编排'上；每多一个agent就多一处可漂移、可泄漏、可失败的接缝

### 最佳实践

- 分层上下文预算：章节prompt分'常驻极简层'(本章必须的一句话人设+当前状态+本章目标)和'按需触发层'(其余设定走关键词/场景触发注入)。每维度设'完整/仅名称'两档+token配额条,默认日常章用'仅名称+本卷章纲',只在关键转折章升级'完整+最近N章'(抄马良写作+AI Dungeon)
- 把'防剧透可见性'与'上下文配额'拆成两个正交机制：可见性是确定性红线(未到开放卷的设定无论如何不进context),不要靠提示模型'别剧透'。这把'AI借角色之口说破幕后黑手'从概率问题变成确定性问题
- 一致性靠'外部结构化记忆+生成后强制审计',不靠'更大上下文窗口'：维护一个知识图谱/canon(角色当前位置/等级/关系/已得物品/已交代信息),每章写前查档、写后抽取本章实体与canon比对,不符则重写
- craft知识与每本书状态分离：anti-slop词表、anti-pattern清单、文风规则放进skill本体(可复用);voice/world/characters/outline/canon/state放进每本书目录(autonovel master/branch模式)
- anti-slop分两层先跑便宜的：机械层用正则扫AI痕迹词(光明的尾巴式强行升华、论文腔连接词、堆砌形容词、残留指令符号),无需LLM;再用LLM层判结构级痕迹(场景跳跃、节奏拖沓、千篇一律的爽点节拍)
- 固定可执行Persona统一文风,注入位置固定：人设简介放系统提示最前(最高指引)+任务提示居中+参考素材放末尾用分隔符包裹。Persona写成可执行指令('每段不超过三行、多动作描写、拒绝空洞形容词')而非'写精彩点'
- 章节生成走'短生成+人确认'的scene-beat循环,而非一次性几千字：给一两句beat→AI写几百字→人校验→下一beat(NovelCrafter模式),把人的judgment插在每个beat而非整章之后
- 严格指令/素材分区防泄漏：设定与前文用代码块/XML标签包裹,末尾加确定性后缀强约束'只输出正文,不输出解释/标签/元信息',生成后正则清洗任何残留指令符号
- 上游改设定要触发下游传播债务预警：改了主角名/世界观,系统列出受影响的大纲/章节(KKKenChow冲突预警+state.json propagation debt),配全局查找替换,而非默默不同步
- 内置'场景转换连接句生成'治AI的硬跳跃,内置'去光明尾巴/去论文腔'后处理——这些是过平台AI检测的commercial-grade硬门槛,不是可选项

### 反模式

- 十agent/三十三维审计式过度工程(novelix)：把'强制一致性'这个对的目标用10个串行agent+7truth文件+33维审计+22改写规则实现,接缝多、脆弱、难用、难调,被一线作者评为overengineered。每个agent都是一处可漂移可泄漏可失败的接缝
- 每章灌整本设定集(whole-bible dump)：token爆炸+成本飙升+prompt臃肿导致指令泄漏进正文+模型注意力稀释反而更易漂移。Sudowrite/NovelCrafter式'存了但要人工每章回灌'就是这个坑的半成品形态
- 把一致性当纯reference problem(存储+注入+信任模型用对)而不做active enforcement：这是所有商业工具6万字后必崩的共同根因。存了设定不等于执行了设定
- 过度模板化→同质化+AI味+被平台拒签：笔灵/蛙蛙式300+生成器堆出来的正文套路感重、语言趋同。模板适合冷启动脚手架,不适合连载正文引擎
- '换更大上下文窗口就能解决吃书'的迷思：窗口再大也有上限,且全书塞进去又贵又慢又稀释注意力。Kimi式长上下文只擅长'读懂'不擅长'连续创作'
- 章节prompt写得又长又满指令：指令与数据混合时越长越容易把约束/设定当素材写进正文(《天命使徒》'提示词长了不行短了也不行';已发表AI文残留指令符号被一眼鉴AI)
- 让AI连续自动续写不设人类节点：4-5章/约20万字后逻辑崩、人设飘、剧情驴唇不对马嘴(毛志慧亲述20万字临界点)。全自动续写=必崩
- '一键500万字全自动'营销话术当真：唐库官方都自承长篇仍需大量人工润色、AI味过重、只在3万字短篇尚可。把automation边界吹到正文全自动是反模式
- 让AI主导剧情走向/节奏/拉期待感：编剧实测'AI擅长讲精彩故事却不会拉期待带节奏','让没情绪的AI写剧本相当于让瞎子搞摄影'。节奏和爽点钩子是human-only区
- 改了上游设定却让下游默默不同步(改主角名但大纲/章节里旧名还在),不做传播债务预警与全局替换——长篇高频自相矛盾来源

### 自动化判定

- **环境/场景描写、旁白、细节扩写** = [automate] — 一线实践反复确认这是AI最稳的强项(《天命使徒》细节描写交给AI;番茄AI扩写/AI续写也定位于此)。确定性代码给定beat和环境要点,AI产出描写
- **起名、设定卡/素材生成、黄金前三章脚手架** = [automate] — 模板化生成器在冷启动阶段高效且低风险(笔灵300+生成器、蛙蛙47类节点脑暴),属于'还没写正文之前把项目立住'的脚手架,可全自动
- **词级anti-slop扫描与残留指令符号清洗** = [automate] — 机械正则即可,无需LLM,便宜确定(autonovel的mechanical immune system)。清光明的尾巴、论文腔连接词、残留提示词符号是过平台检测的硬门槛
- **章节正文生成(单beat/短段)** = [semi-auto] — AI出草稿,人按scene-beat校验(NovelCrafter模式)。一次性几千字全自动必然漂移;短生成+人确认是可控区
- **canon/设定实体抽取(Observer/Reflector式)** = [semi-auto] — 抽取可自动(每章抽人物/地点/物品/状态delta),但关键事实需人确认入canon——抽取本身有错误率,错的canon比没canon更糟
- **一致性审计(对照canon)** = [semi-auto] — 机械层(数值/物品/位置矛盾)可全自动;结构/人设/情绪一致性需LLM判断+人最终拍板。这是与所有商业工具的关键差异,但LLM审计有漏判,不能纯自动
- **三级大纲(总纲/卷纲/章纲)搭建** = [semi-auto] — AI可生成大纲草稿,但arc走向、冲突密度、卷与卷的承接由人决定。蛙趣拼文/马良的三级大纲是人机协作产物
- **剧情走向、转折设计、爽点钩子、拉期待感** = [human-only] — 编剧/作者一致认为AI'会讲故事不会带节奏';求是网点出AI只能机械刺激不能情感打动。剧情主导权交给AI=同质化+无灵魂+被拒签
- **节奏把控(章节节奏、卷节奏、付费卡点)** = [human-only] — 商品内容内核是用期待感和情绪带动需求,'让没情绪的AI写剧本相当于让瞎子搞摄影'。节奏是网文商业价值核心,必须人类主导
- **防剧透可见性设置(哪些设定按第几卷开放)** = [human-only] — 系统只能确定性地执行'未开放不进context'这条红线,但'哪条设定何时揭晓'是叙事悬念设计,属作者意图,必须人来设(马良写作的可见性是手动按卷勾选)

## 调研流：去AI味 + 中文创意写作的模型选择（A. AI味成因与去除；B. 网文创作模型横向对比与分工）

**摘要**：中文"AI味"是一组可枚举的语言指纹（工整排比、"不是A而是B"、破折号/冒号堆叠、形容词堆砌名词缺失、情绪直白告知、说教升华腔），但最关键的结论是：句子层去AI味（换词/换句/调标点）只能骗过句子层，骗不过叙事层——马里兰+DeepMind的StoryScope仅用30个叙事特征就能93.2%区分人/AI，说明AI味的根在RLHF对齐导致的"叙事趋同/mode collapse"，prompt救不了根（但Verbalized Sampling等可部分缓解）。对pipeline最致命的设计含义：去AI味必须在结构/叙事层动刀并作为独立审查pass，且模型选择本身就是去AI味的一部分。用户的核心假说成立但需精确化——为agentic/coding/reasoning优化的训练确实与"按硬约束写正文"负相关（NeurIPS 2025证明CoT让14个模型中13个指令遵循变差），所以正确架构是"分角色多模型"：reasoning/agentic模型做大纲/逻辑/一致性校验，强中文文学性模型（Claude正文、豆包/GLM/Qwen网文在地化）关掉thinking写正文，剧情走向与节奏由人类拍板。所有评测（尤其EQ-Bench）都被"堆砌文学手法"reward-hack，必须实测>评测。


### 关键发现

- **[high]** 中文AI味是一组可枚举、可自查的语言指纹，而非玄学；至少包含11类高频模式可直接写成检查清单
  - 依据：维基百科AI写作迹象清单（否定平行结构Not-only-X/虚假范围From-X-to-Y/说教免责'值得注意的是'/模糊归因'批评者认为'/三段式排比/僵化结尾'综上所述'）；翔宇8特征（句式灵动不足=读起来像念经、词汇模板化、逻辑过度规整、语气一本正经、情感扁平、修辞公式化'不仅A而且B'、细节缺失=全是形容词没有名词、过度完美）；Tenten四支柱（对比句式/排比三段式/反问设问/破折号补充）
  - 出处：https://www.managertoday.com.tw/articles/view/71293 ; https://xiangyugongzuoliu.com/ai-style-writing-8-common-giveaways ; https://tenten.co/learning/tenten-maga-prompt-fix-ai-tone
  - 启示：pipeline可内置一份确定性的'反AI味检查清单'作为正文产出后的机检/审查pass，逐条扫描这些模式并触发改写；这是少数能机器化的去AI味环节
- **[high]** 句子层去AI味（换词/换句/调标点）只能改善表面，改不掉更深的'叙事指纹'——这是prompt救不了、必须在结构层处理的部分
  - 依据：马里兰大学+Google DeepMind的StoryScope：仅用30个叙事特征（不看用词/句式/标点）就能以93.2%准确率区分人类与AI写作，加入风格线索后准确率仅再高<3%；五大叙事差异：AI叙事者77%直接点明主题（人类52%）、AI对话59%出现哲学讨论（人类34%）、AI偏好单线叙事、回避道德模糊、引用全是模糊暗指（72%）；所有AI模型在叙事空间聚集于同一共享区域，Ethan Mollick：'要求AI用不同风格写作几乎改变不了叙事层差异'
  - 出处：https://www.tmtpost.com/8008644.html (StoryScope, 马里兰大学+DeepMind, 60000+小说)
  - 启示：去AI味的主战场在大纲/叙事结构（让人来定走向、留白、不点题、引入道德模糊与多线），而不是只做句子润色；任何只在文本层'降AI率'的工具都是治标
- **[high]** AI味的深层根因是RLHF/对齐导致的'多样性坍缩(mode collapse)'，prompt无法治本，但Verbalized Sampling等prompt级技术可部分恢复
  - 依据：多篇论文证实对齐降低输出多样性：ICLR 2024 'RLHF substantially decreases diversity'、Kempner Institute NAACL 2025 'alignment reduces conceptual diversity'、Gwern mode-collapse综述（'AI slop/ChatGPTese'）；Verbalized Sampling(arXiv 2510.01171)：直接prompting在DPO后多样性从20.8%跌到10.8%，VS恢复到base的66.8%、创意写作多样性提升1.6-2.1×、人评+25.7%，且training-free只改prompt
  - 出处：https://arxiv.org/html/2510.01171v1 ; https://proceedings.iclr.cc/paper_files/paper/2024/file/5a68d05006d5b05dd9463dd9c0219db0-Paper-Conference.pdf ; https://gwern.net/doc/reinforcement-learning/preference-learning/mode-collapse/abstract
  - 启示：指望调高temperature恢复多样性是错的（高温反而触发退化）；正文发散环节应引入Verbalized Sampling式采样（让模型先列多个候选方向再选），并优先用对齐较弱/文学性强的模型
- **[high]** 每个模型有改不掉的'叙事指纹'，所以模型选择本身就是去AI味的一部分——靠prompt抹平指纹无效，应'扬长避短分工'
  - 依据：StoryScope六类作者归属准确率68.4%：GPT频繁'梦境转折'、Claude'波澜不惊像白开水'、Gemini'每个角色都从外部描述像人物档案卡'；maplefeather实测：Claude太礼貌太平顺冲突感被磨掉、DeepSeek修辞浮夸煽情做作、Grok画面感盖过逻辑
  - 出处：https://www.tmtpost.com/8008644.html ; https://maplefeather.com/article/ai-writing-novels-test
  - 启示：pipeline不要试图把一个模型'调'成全能；而要让不同模型干它指纹最不碍事的活（如Gemini做设定、Claude做对话/伏笔、避免让GPT写需要克制的情绪戏）
- **[high]** 用户'agentic/coding模型牺牲创意写作'的假说成立，但精确机制是：reasoning/CoT训练与'按硬约束写正文'负相关，而非泛泛地'变笨'
  - 依据：NeurIPS 2025 'How Reasoning Harms Instruction-Following'：开CoT后14个指令微调模型中13个IFEval变差，Llama3-8B-Instruct 75.2%→59.0%；失败模式=过度关注内容、漏掉简单规则（如字数）、加冗余'贴心'内容违反约束；IBM：reasoning模型在可验证域(代码/数学)训练评测，creative writing研究极少；'Thinking LLMs'论文：直接prompt模型写思考过程反而hurts非数学任务
  - 出处：https://www.linkedin.com/posts/omarsar_new-paper-how-reasoning-harms-instruction-following-activity-7330227896717824007-8hP0 ; https://www.ibm.com/think/topics/reasoning-model ; https://arxiv.org/html/2410.10630v1
  - 启示：架构上把'思考'和'写正文'分离：reasoning/thinking模型用于大纲推演/逻辑校验/一致性检查；写正文时关闭thinking或用非推理模型——否则它会为了'想清楚'而违反字数/文风/不泄漏prompt等硬约束（这恰好也是prompt泄漏到正文的诱因之一）
- **[medium]** prompt泄漏到正文的根因是LLM无法从token流区分'指令'与'内容'，长prompt会放大该故障；解法是结构隔离，不是加更多'不要泄漏'的禁令
  - 依据：IBM/OWASP LLM01：'系统提示和用户输入都采用相同格式：自然语言，LLM无法仅根据数据类型区分指令和输入'；防御最佳实践=分离指令区与内容区、用content-slot模板（用户内容只进内容槽不进指令槽）、监控输出剥离系统级工件；安全内参实测：宽泛的'任何情况下不泄漏'防护<5%，结构化/职责化加固才有效
  - 出处：https://www.ibm.com/cn-zh/think/topics/prompt-injection ; https://evvolabs.vn/prompt-injection-attacks-zh ; https://www.secrss.com/articles/82209
  - 启示：写正文的章节prompt要做到：①约束/context区与'请输出正文'区用XML标签或分隔符硬隔离；②每章prompt尽量短（约束放进system/常驻文件，章节prompt只给本章必要信息）；③产出后用确定性后处理剥离任何混入正文的指令痕迹/元注释；④用'控制token'契约让模型只在正文区输出
- **[high]** 长篇连续续写'4-5章后崩'是工程问题（上下文溢出/记忆缺锚/风格漂移），不是模型文笔问题；不做上下文管理通常20-40万字开始密集出现设定矛盾
  - 依据：马良写作内部测试：纯长对话续写在20-40万字区间开始密集出现设定矛盾，启用结构化框架后同篇幅几乎不再出现角色状态错乱；崩塌三源=上下文溢出（旧信息被悄悄截断）、记忆缺锚（无外部大脑记录角色当前状态）、风格漂移（每次重新交代文风不一致）
  - 出处：https://maliangwriter.com/blog/ai-write-long-novel-framework

### 最佳实践

- 模型分工方案(核心)：①结构/大纲/逻辑推演/一致性校验位→reasoning/agentic模型(DeepSeek R1/GPT o3/Gemini，可开thinking)；②正文生成位→强中文文学性模型且关闭thinking(精华章Claude Opus/Sonnet；网文批量章豆包/Qwen/GLM)；③去AI味与连贯性审查位→独立subagent。用client注入(base_url+model)让每个位可热插拔、可盲测
- 选型用'对标章盲测'而非榜单：把同一份章纲喂候选模型，盲读前500字(代入感)+第3章(是否崩)+章末钩子力度，自己评，因为EQ-Bench/LMArena都被'堆砌文学手法'reward-hack(与专家判断仅43%一致)
- 正文环节默认关thinking：reasoning/CoT会为'想清楚'而违反字数/文风/不泄漏prompt等硬约束(NeurIPS证明14模型中13个指令遵循变差)；要思考就放到大纲阶段
- 章节prompt结构隔离：约束/context常驻于system或外部设定文件，章节prompt精简且用XML标签/分隔符把'约束区'与'正文输出区'硬分开，正文只进content-slot；产出后确定性剥离混入正文的指令痕迹/元注释——这是治prompt泄漏到正文的根
- 去AI味做两层：句子层=确定性清单pass(否定平行/虚假范围/三段排比/说教升华/破折号冒号/形容词堆砌名词缺失)+禁破折号冒号逼出口语句式+情绪词降级；叙事层=在大纲阶段由人引入留白/不点题/道德模糊/多线，因为叙事指纹prompt改不掉
- 正文发散用Verbalized-Sampling式采样(让模型先列N个候选走向/写法再选)以对冲mode collapse，而不是调高temperature(高温反而触发退化)
- 风格锚定每章重注入对标文风样本+独立改写pass，不可一次注入全书不管(长文必漂移、且会塌缩到单一风格)；对'统一全书文风'考虑专用文风迁移微调(StyleLLM/TinyStyler路线)
- 长篇连贯靠工程栈不靠长上下文：三级大纲(总纲→卷纲→章纲→正文)+知识图谱/RAG兜底+角色状态快照(含knows/does_not_know/reader_knows_but_character_does_not)+按卷渐进可见性(防剧透)+草稿与正式章分离(人工确认才入记忆库防污染)
- 把爽感编码成章纲结构化字段：爽点密度(每~300字一小冲突)、章末钩子、憋屈-打脸-收获循环、情绪守恒(憋屈时长∝打脸力度)、打脸间隔≤3章——密度可机检，但走向/力度由人拍板
- 成稿是人机协作出版：AI出80分初稿，人推到95(补人物口吻/必要的难听话/尴尬停顿/伏笔回收)；明确'AI主导生成，人类主导审美'的边界

### 反模式

- 完全放手让AI连续续写——4-5章后崩是结构性必然(上下文溢出+记忆缺锚+风格漂移)，20-40万字必密集吃书；不上记忆工程栈，换再强的模型也救不了
- 只在句子/词/标点层'降AI率'——StoryScope证明改词换句调标点是表面功夫，93.2%的AI味来自叙事结构(说教/单线/回避道德模糊)，文本层工具治标不治本
- 用reasoning/thinking模型写有硬约束的正文——CoT会漏掉简单规则(字数)、加冗余'贴心'内容、并把约束/思考混进正文(正是prompt泄漏诱因)
- 章节prompt写得又长又把约束和正文要求混排——指令与内容同为自然语言token流，LLM无法区分，长prompt放大泄漏；且用宽泛的'任何情况下不要泄漏'禁令(实测防护<5%)
- 照搬EQ-Bench/LMArena排名选模型——被堆砌比喻/文学手法reward-hack，弱模型登顶；不做自己的对标章盲测
- 一次性把全书章纲生成完——长篇是边写边长，过早锁死章纲会在剧情自然生长时束手束脚(卷纲先行、章纲滚动生成才稳)
- 指望调高temperature恢复多样性——高温触发退化(短语重复/降级)，对齐导致的mode collapse要靠采样策略(VS)和换弱对齐/文学性模型
- few-shot喂一次样章就当全书文风搞定——长文必漂移，且模型会塌缩到样本里的某一种风格、产物仍被AI检测器识破
- 把剧情走向/节奏/爽点力度交给AI拍板——AI擅长结构一致性而非'创造火花'，且叙事会趋同/爱说教爱单线；走向与节奏是人类主导区
- 用同一个模型干所有活——强agentic模型(如GLM-5系/GPT o3)写正文易牺牲文学性，强文学模型做一致性校验又慢又贵；不分工=让专才做通才的事

### 自动化判定

- **环境描写、旁白、细节扩写** = [automate] — 一线实践与多源均证实AI在此最好用；属低风险、可由文学性模型直接产出
- **句子层去AI味润色(清单扫描+标点禁令+情绪词降级)** = [automate] — 反AI味模式可枚举、可确定性机检；独立subagent审读效果已验证(GLM自审'背后发凉→愣了一下')
- **一致性校验/伏笔追踪/吃书检测/时间线冲突** = [automate] — 属确定性可验证域+RAG/知识图谱，正是agentic模型强项；QMAI六维审查/连贯性Lint已落地
- **大纲推演(总纲/卷纲/章纲生成)** = [semi-auto] — AI基于上位约束推演卷纲/章纲高效，但走向/转折/锁定需人拍板；章纲滚动生成而非一次锁死
- **世界观/力量体系/角色卡构建** = [semi-auto] — AI能批量产出设定草案(创意位用Claude Opus)，但核心设定与卖点需人审定以防同质化
- **叙事层去AI味(留白/不点题/道德模糊/多线)** = [semi-auto] — 叙事指纹prompt改不掉(StoryScope)，须在大纲阶段由人主动引入'非AI'的叙事选择，AI执行
- **风格锚定/全书文风统一** = [semi-auto] — few-shot长文必漂移且塌缩单一风格；需每章重注入+独立改写pass，必要时专用文风微调，人监控漂移
- **正文成稿(初稿→出版稿)** = [semi-auto] — AI出80分初稿，最后95分(人物口吻/必要难听话/尴尬停顿/伏笔回收)需人工，多源一致结论
- **章末钩子/断章设计** = [semi-auto] — 套路化钩子可生成，但力度与悬念点选择影响追读率，需人调
- **剧情走向、节奏把控、爽点力度拍板** = [human-only] — 用户一线约束+ACM/Jenova研究一致：AI叙事趋同、爱说教、回避冲突，'创造火花'与节奏是人类主导区

### 存疑

- 任务列出的'Fable'模型身份与创作能力数据未能核实——在用户配置中Fable作为Anthropic系触发词出现，疑为Claude家族别名，但无公开创意写作基准可引用(⚠️推断)，需用户澄清或单独实测
- GLM-5系/豆包等'agentic融合'新版相比上一版的创意写作能力是退步还是进步，缺纵向对照实测；'coding bias牺牲文学'在国产模型上是否同样成立需自测验证
- Verbalized Sampling在中文网文长篇正文上的实际去AI味/爽感效果未见实测，仅有英文创意写作通用数据
- 专用文风微调(StyleLLM/TinyStyler/LoRA) vs 纯prompt few-shot 在'统一百万字文风'上的ROI与工程成本对比缺数据
- prompt泄漏到正文的最优'结构契约'(XML标签/分隔符/control-token/content-slot的具体组合)缺公开实证基准，需在目标模型上A/B
- StoryScope揭示的'叙事指纹'能否通过工程化的'反指纹大纲规则'(强制多线/道德模糊/不点题/明确引用)被规避到检测器之下，还是属于训练范式固有、只能靠人类深度介入——尚无定论
- 各模型最佳'关thinking写正文 vs 开thinking写大纲'的切换点与温度/min_p参数在中文网文上的最优配置缺系统调参数据

## 调研流：多agent编排 + 章节级prompt工程 + 审校rubric（AI写网文章节生产循环）

**摘要**：百万字连载"4-5章必崩"的根因不是模型弱，而是缺少把LLM当子程序、用确定性代码当driver的生产循环：现有一线实践（Trae Novel-Skills、xindoo/Sumeru 320章实测、KKKenChow/ai-novel-writer、马良写作）和学术工作（Re3、StoryWriter、DOC/RecurrentGPT）高度收敛于同一套骨架——分层状态文件作单一事实源 + 确定性"单章prompt编译器"只抽取本章最小context + writer/reviewer/continuity 三类独立调用 + 改稿循环≤3轮 + 每章后增量更新state。prompt泄漏进正文的根因是"长prompt里指令与数据没有结构分离"，修复手段是XML分区把设定标成"数据非指令"、用代码把"只输出正文"硬约束作为后缀拼接、并把context压到最小。节奏/剧情走向是必须人类当driver的部分（DeepSeek"八千字才觉醒"证明模型默认节奏是错的），环境描写/旁白/扩写可全自动。审校必须是独立于writer的打分调用，按加权rubric量化（人设/世界观/时间线为硬门，爽点/钩子/AI味为加权分），阈值判定由确定性代码做、循环改稿直到通过。AI味需按"看密度不看出现"的语体感知规则去除，而非一刀切。


### 关键发现

- **[high]** 章节生产必须是'确定性代码当driver、LLM当被包裹的子程序'的闭环循环：planner→outliner→prompt-compiler(确定性)→writer→reviewer→continuity-checker→style-editor→state-updater，而不是让模型连续自由续写。多个独立一线项目与论文收敛到这同一骨架。
  - 依据：Trae Novel-Skills：novel_coord统筹 + novel_world/plot/char/writer/checker 6子代理 + 循环修正≤3轮 + '只读取最近3章' + '唯一真相源机制'；xindoo/Sumeru 把流程拆成 worldbuilder/topic/outline/write/review/polish/finalize 7模块并实测产出320章100w+字；Re3 把人类写作分解为 Plan-Draft-Rewrite-Edit 四模块递归重提示。
  - 出处：https://forum.trae.cn/t/topic/18011 ; https://cloud.tencent.com/developer/article/2664367 ; https://arxiv.org/abs/2210.06774
  - 启示：pipeline 的顶层是确定性循环脚本，每个角色一个明确输入/输出契约；模型每次只被调用做一件窄事，调用完代码立刻收回控制权做校验和state更新——这就是把'4-5章崩'转成'可稳定推进'的结构。
- **[high]** '单章prompt编译器'必须是确定性检索+拼装步骤、不能是LLM agent；它从世界观/人物/剧情线/时间线/滚动摘要里只抽取本章需要的最小context，这是同时解决'长prompt泄漏进正文'和'设定漂移'的关键节点。
  - 依据：KKKenChow/ai-novel-writer 的章节生成='分类上下文构建 + 六阶段渐进节奏控制 + 四级剧透过滤，避免Token膨胀和剧情前移'，RAG为'核心设定直取 + 相关大纲提取 + 前章摘要 + 语义搜索补充'；Trae'只读最近3章'；StoryWriter 用 Coordinator agent 动态压缩历史'只保留与当前事件相关的events和characters'；Re3 prompt里只放'Current Section Outline'+把上一段原文作为'Autoregressive Context'。
  - 出处：https://github.com/KKKenChow/ai-novel-writer ; https://arxiv.org/pdf/2506.16445 ; https://arxiv.org/abs/2210.06774
  - 启示：编译器=代码控制的retrieval，决定'哪几条canon+哪几个在场角色+最近N章摘要+本章beat+禁止剧透清单'进prompt。context越小、越结构化，越不会把设定当正文输出，也越不会因塞太多无关信息稀释注意力。
- **[medium]** prompt泄漏进正文的根因是'长prompt里把约束/设定和要写的正文混在同一段、没有指令/数据边界'；修复是结构分离：用system/developer放硬约束、用XML分区把设定标注为'数据非指令'、用代码把'只输出小说正文'作为后缀强制拼接、并把context压到最小。
  - 依据：Gemini官方提示策略明确用 <context>[Insert User Input Here - The model knows this is data, not instructions]</context> 把上下文标为数据，并把 role/instructions/constraints/output_format 分区；practitioner反复观察长prompt会被模型'当正文续写'；Trae/KKKenChow 都用最小context+分层文件规避token膨胀。
  - 出处：https://ai.google.dev/gemini-api/docs/prompting-strategies?hl=zh-cn ; https://www.promptingguide.ai/zh/prompts/adversarial-prompting/prompt-leaking
  - 启示：单章prompt模板必须做到：设定全部在带标签的<reference>块且声明'仅供参考、不得复述'；'只输出正文、不要输出大纲/解释/设定/元叙述'是代码拼接的硬invariant而非'希望模型记住'；writer调用绝不携带整本书的meta指令。⚠️推断成分：'混排导致续写'的精确机制属经验归纳。
- **[high]** 节奏和剧情走向必须人类当driver，环境描写/旁白/扩写可全自动——这与practitioner实践一致，且模型在无约束时默认节奏是错的，所以节奏要靠确定性的beat预算控制而非交给模型自由发挥。
  - 依据：桦一聊AI实测：DeepSeek'描写功力最佳……但叙事像老火汤，八千字才等到系统觉醒'，四平台编辑拒稿理由一致是'文笔细腻但节奏拖沓'；Sumeru原则C：'并行化只做体力活，不要做核心决策……世界观底层设定、主线节奏不适合多Agent投票'；蛙蛙写作'20章后易出现战力崩坏或人设偏移，需人工干预'。
  - 出处：https://aitool.csdn.net/686211b78e4d0a6c534cdad1.html ; https://cloud.tencent.com/developer/article/2664367 ; https://www.hello123.com/wawawriter
  - 启示：节奏=outline层的'每章必须推进的情节点数/爽点位置/钩子'，由人定beat、代码校验密度；writer只负责把一个已定死的beat sheet扩写成正文。把'写多少字到哪个剧情节点'写进章纲，模型就不会拖到八千字。
- **[high]** 审校必须是独立于writer的打分调用，按加权rubric量化：人设/世界观/时间线一致性设为'硬门(0违规)'，爽点兑现/钩子/AI味/节奏设为加权分；总分≥阈值且硬门全清才通过，否则带具体findings改稿、循环≤3轮。
  - 依据：SkillHub novel-writer 章节质检：逻辑一致性25%/钩子20%/节奏20%/敏感词20%(0违规)/字数15%，总分≥80合格、敏感词>0绝不发布；snowflake-fiction 评分维度：冲突强度25/情绪密度25/期待感20/节奏控制15/钩子设计15，并有独立'开篇检查报告'；Trae'三重校验+最多3轮自动修正'。
  - 出处：https://skillhub.cloud.tencent.com/skills/novel-writer-agent ; https://github.com/hestudy/snowflake-fiction ; https://forum.trae.cn/t/topic/18011
  - 启示：reviewer输出结构化JSON{各维分数, violations[], verdict:PASS|REVISE}+一个裸sentinel(PASS/REVISE)；确定性代码对sentinel容错解析后branch，不让模型自己决定'够不够好'。'让模型给自己打分通过'是反模式，必须是单独一次调用甚至单独agent。
- **[high]** 连续性校验应是独立agent/调用，吃的是canon/时间线/人物文件而非写作context，并尽量工具增强（角色档案查询、时间线一致性检查、entailment式事实核对），输出违规清单而非自由文本。
  - 依据：Qwen3-14B方案用Function Calling：get_character_profile / check_timeline_consistency / compare_writing_style，'写关键剧情前先查主角性格档案'；Re3 的Edit模块用entailment做'长期事实一致性'检查；YILING0013/AI_NovelGenerator 有独立 consistency_checker.py；apiad多agent博客把failure归为 Character Amnesia/Plot Holes/Structural Breakdown，需专门架构兜。
  - 出处：https://adg.csdn.net/69730db7437a6b40336b6bb0.html ; https://arxiv.org/abs/2210.06774 ; https://github.com/YILING0013/AI_NovelGenerator ; https://blog.apiad.net/p/ai-driven-storytelling-with-multi-3ed
  - 启示：continuity-checker与quality-reviewer分开（前者查'对不对'，后者评'好不好'）。把'查角色当前状态/时间线'做成对state文件的工具调用，比让writer自己记住可靠得多；命中canon冲突时上报、由人裁决。
- **[high]** state必须是分层文件作单一事实源，且每条设定打'Canon四态'标签；冲突铁律是显式上报+三选项(改新想法/改旧设定/把冲突变成剧情冲突)，绝不默默采纳或丢弃——默默处理是长线漂移的头号成因。

  - 依据：StoryWriter 的 Coordinator '动态压缩历史只保留相关events/characters'；Re3 在prompt里保留上一段verbatim作'Autoregressive Context'；snowflake-fiction 每章后输出'内容质量评估报告'再决定是否进下一步；Sumeru '中间数据必须持久化(.sumeru/)、可复用可回滚可续跑'。

  - 出处：https://maliangwriter.com/blog/ai-model-selection-guide ; https://aitool.csdn.net/686211b78e4d0a6c534cdad1.html
  - 启示：pipeline应让LLM client可注入(base_url+model可换)，把模型当可替换后端：高潮章用强文笔模型、日常章用性价比模型、reviewer/continuity可用不同模型交叉验证。具体哪个模型好由rubric打分决定，不写死。
- **[medium]** '黄金三章/爽点密度/付费卡点'等网文craft是可参数化进outline的硬约束，不是模型能自发产出的；爽点要按间隔排布、章末必留钩子、禁止水文。
  - 依据：snowflake-fiction百万级规划：爽点密度'每3章小爽/每10章中爽/每30章大爽/每100章超大爽'+番茄30章付费卡点+多阶段地图升级；SkillHub:'每章结尾必须留钩子、禁止水文(每1000字至少1情节点)、前10章快节奏'；作家助手'黄金三章不是固定格式但必须尽快立卖点'；起点社区:番茄14天给量周期+静默期决定更新策略。
  - 出处：https://github.com/hestudy/snowflake-fiction ; https://skillhub.cloud.tencent.com/skills/novel-writer-agent ; https://m.write.qq.com/portal/article/detail?CAID=12871234803277101 ; https://www.qidianclub.com/threads/428
  - 启示：剧情线.md里要有结构化的'爽点排布表'和'钩子要求'，prompt-compiler把'本章要兑现的爽点+结尾钩子类型'塞进章纲，reviewer按此核对兑现与否。craft变成可校验的rubric项，而不是指望模型'懂网文'。

### 最佳实践

- 顶层用确定性循环脚本当driver：for each chapter → compile_prompt()(代码) → write()(LLM) → review()(LLM, 独立) → check_continuity()(LLM/工具, 独立) → if not PASS: revise()循环≤3轮 → style_edit()(LLM) → update_state()(代码包裹的抽取)。每个LLM调用是子程序，调用后代码立刻收回控制权做校验和写state。
- 八个角色的输入/输出契约写死：planner(输入:剧情线.md+人类决策→输出:本arc beats)、outliner(输入:arc beats+state→输出:单章beat sheet JSON)、prompt-compiler(输入:state文件+章纲→输出:分区好的单章prompt, 纯代码)、writer(输入:prompt→输出:纯正文)、reviewer(输入:正文+章纲+rubric→输出:打分JSON+sentinel)、continuity-checker(输入:正文+canon/时间线/人物→输出:violations[])、style-editor(输入:正文+AI味规则→输出:润色稿)、state-updater(输入:批准稿+旧state→输出:state delta)。
- 单章prompt编译器只用确定性检索装配，绝不是LLM agent：抽取规则=核心设定直取(本章在场角色的canon)+相关大纲提取(本章beat)+前章压缩摘要+上一章末段原文+语义搜索补少量相关伏笔；上限'只读最近3章+本章相关canon'，宁可少不可多。
- prompt结构强制三分离：(1)system/developer放角色+硬invariant；(2)user里所有设定放带标签的<reference_*>块并显式声明'仅供参考，禁止在正文中复述或解释'；(3)'只输出本章小说正文，不要输出大纲/设定/解释/标题/元叙述'由代码作为最后一行强制拼接(SUFFIX), 不依赖模型记忆。
- 节奏/剧情走向/选题/canon裁决保持人类当driver：LLM只在这些环节提供'三套选项'供人拍板(对齐Canon冲突三选项机制)；把'本章必须推进到哪个情节点+字数预算+爽点位置+钩子类型'写进章纲当硬约束，模型不得自行决定节奏。
- 审校与写作必须是两次独立调用：reviewer吃rubric给量化分(人设/世界观/时间线=硬门0违规, 爽点/钩子/AI味/节奏=加权分), 总分≥80且硬门全清才PASS；不通过则把具体violations回灌给writer做定向改稿, 循环≤3轮, 第3轮仍不过则flag人工。
- continuity-checker独立且工具增强：把'查角色当前状态/查时间线/查伏笔状态'做成对state文件的检索调用(get_character_state/check_timeline/list_open_foreshadowing), 让校验基于文件事实而非writer记忆；命中canon冲突→显式上报+三选项, 由人裁决。
- state五文件+Canon状态层：世界观.md(canon事实+glossary唯一写法+力量体系)、人物.md(每角色:身份/voice/关系/当前状态字段/角色弧)、剧情线.md(主线beats+伏笔账本open/closed+爽点排布表, 人主导)、时间线.md(事件→章→故事内时间→地点→在场人物)、滚动摘要.md(每章一句话+最近N章压缩摘要+上一章末段原文)。每条设定打Canon/Pending/Rejected/Idea Bank。
- state增量更新而非覆盖：每章PASS后state-updater从正文diff出新事实, append写入, 角色'当前状态'字段增量改, 时间线append事件, 伏笔set/resolved标记；canon晋升需人/规则确认；任何与已有canon冲突一律显式flag。
- AI味按语体感知去除(看密度不看出现)：网文叙事保留气势/铺垫/合理修辞, 只打击固定频率反复的指纹(否定平行/三段式排比/虚假范围/说教式免责/比喻链铺满/加粗总结/万能套话)；正向手段=长短句交替+具体名词数字+show-not-tell+情感注入；可把qu-ai-wei的51类模式接进reviewer的AI味维度。
- LLM client可注入、模型可换后端：高潮章用强文笔模型(如Claude系)、日常章用性价比模型(如DeepSeek/Gemini Flash/Qwen)、reviewer/continuity用与writer不同的模型做交叉验证；temperature分场景(高潮0.5稳, 日常0.8活)。具体选型由rubric在自己题材上A/B决定, 不写死模型ID。
- 中间数据全程持久化、可断点续跑：state文件+每章稿+评分报告落盘(参考Sumeru .sumeru/), 任何一章可回滚重跑, 循环可中断后恢复——长篇连载是跨数月多session工程, 不能依赖单次会话内存。

### 反模式

- 让模型连续自由续写整本书——4-5章后逻辑/人设/世界观必崩(蛙蛙写作20章后战力/人设崩坏可证)；没有state回灌和校验的续写必然漂移。
- 把整本书的设定集、长篇约束和'要写的正文'塞进同一个超长prompt——模型会把约束/设定当正文续写出来(prompt泄漏)，且无关信息稀释注意力。
- 用LLM agent来做'单章prompt编译/context抽取'——retrieval装配必须是确定性代码控制, 交给模型会导致context范围不可控、又长又泄漏。
- 让writer同时负责自评/自查一致性——既稀释写作质量, 又因为'自己给自己打分'天然偏松；审校和连续性校验必须是独立调用甚至独立模型。
- 把节奏和剧情走向交给模型自由发挥——DeepSeek'八千字才系统觉醒'被四平台编辑拒稿, 证明模型默认节奏是错的；节奏必须是outline层的硬约束。
- 对'多Agent投票'式地决定世界观底层设定/主线节奏等核心创意决策——Sumeru原则C明确这类决策不适合并行投票, 应人类当driver。
- state用整体重写覆盖而非增量append——容易丢历史事实、回收不了伏笔；冲突时默默采纳新想法或默默丢弃旧设定是长线漂移的头号成因。
- 审校只看'总分'不设硬门——人设/世界观/时间线矛盾是体验崩塌的致命项, 必须是0违规的硬gate, 不能被高爽点分数平均掉。
- 一刀切删除所有排比/成语/修辞来去AI味——会把网文该有的气势和铺垫也删掉(qu-ai-wei警告'过度修正')；应按语体+密度判定。
- 改稿循环无上限或无具体findings——'再改好一点'式空泛反馈无法收敛；必须把reviewer的具体violations定向回灌、且设≤3轮上限+第3轮不过转人工。
- 把模型ID/选型写死并迷信单一评测——'评测≠实践'(用户已实测GLM优于GPT、DeepSeek描写'语言怪')；client要可注入、按自己题材用rubric A/B复测。
- 依赖单次会话上下文窗口记住前情而不落盘——长篇是跨session工程, 不持久化state就无法断点续跑、无法回滚, 一旦中断全崩。
- 把'黄金三章/爽点'当成模型能自发产出的能力——它是可参数化的craft约束, 必须进outline+进rubric核对兑现, 否则模型只会写'散文不是网文'。

### 自动化判定

- **选题/卖点/市场定位** = [human-only] — 市场判断+平台适配是商业决策, 模型可提选项但拍板靠人(Sumeru选题阶段'不是我喜欢而是有没有卖点')。
- **主线/分卷规划(planner)·剧情走向** = [human-only] — practitioner实践证明剧情走向需人类主导；Sumeru原则C核心决策不并行投票。模型只提'三套方向'供选。
- **节奏/爽点排布/付费卡点设计** = [human-only] — 节奏把控目前仍需人拍板(DeepSeek默认节奏拖沓被拒稿)；但具体爽点桥段AI可提议, 人确认密度与位置。
- **单章章纲生成(outliner)** = [semi-auto] — AI按arc beats+state起草beat sheet(目标/冲突/爽点/钩子/字数), 人快速核对节奏后放行。
- **单章prompt编译(prompt-compiler)** = [automate] — 确定性检索+装配, 规则固定(在场角色canon+本章beat+压缩摘要+上一章末段+禁剧透清单), 不需LLM、不需人。
- **正文生成(writer)·环境描写/旁白/扩写** = [automate] — practitioner明确AI擅长环境/旁白/扩写；给定tight beat sheet, AI把beat扩写成正文可全自动。
- **质量打分(reviewer)** = [automate] — 独立LLM调用按rubric出结构化分+sentinel；阈值branch由确定性代码做。是机器oracle不是人。
- **连续性/canon/时间线校验(continuity-checker)** = [semi-auto] — 工具增强检索+entailment可自动跑出violations[]；但命中canon冲突的取舍(改新/改旧/变剧情)需人裁决。
- **定向改稿循环(revise)** = [automate] — 把具体violations回灌writer重写、再过reviewer, 循环≤3轮全自动；第3轮仍不过才flag人工。
- **去AI味/润色(style-editor)** = [semi-auto] — 按密度规则自动改一遍可达~78分(snowflake人语化), 但关键章/高潮章建议人工终审定稿。
- **state增量更新(state-updater)** = [semi-auto] — delta抽取是LLM、append写入是代码全自动；但canon晋升(Pending→Canon)需人或规则确认, 防误固化。
- **canon冲突裁决** = [human-only] — Canon四态机制铁律:冲突显式上报+三选项由创作者拍板, AI不替人拍创意板。
- **模型选型/温度调度** = [semi-auto] — 可按章型自动路由(高潮强文笔模型/日常性价比模型)+温度策略(高潮0.5日常0.8), 但选型基线由人用rubric A/B复测后设定。

### 存疑

- 连续性校验的工具增强到什么程度才够：纯prompt核对 vs 结构化函数调用(get_character_state等) vs 向量库RAG vs entailment模型——哪种在百万字尺度上召回率/成本最优, 缺独立基准, 需自建eval。
- 滚动摘要的压缩比与保真度权衡：每章一句话+最近N章压缩, N取多少、压缩到什么粒度才能既防token膨胀又不丢回收伏笔所需的事实？StoryWriter的Coordinator压缩缺中文网文场景数据。
- 审校用与writer同模型还是异模型交叉验证更划算？异模型能减少'同源盲点', 但增加成本与口径不一致风险, 未见网文场景对比数据。
- 改稿循环≤3轮是经验值(Trae)；定向改稿在第几轮收敛、哪些violation类型靠改稿改不动只能重写或回outline, 需要在真实题材上统计。
- 去AI味改写会不会损伤已通过的一致性/爽点(改着改着把伏笔或人设细节改丢)？style-editor后是否需要再过一遍轻量continuity-check, 成本是否值得。
- 模型实测口径:用户实测GLM>GPT、DeepSeek描写'语言怪', 与厂商评测(Claude文笔最高)冲突——'评测≠实践', 需要用本pipeline的rubric在目标题材上做盲测A/B才能定选型, 当前无可信公开结论。
- 番茄等平台2026收紧AI审核(下架AI低质内容、49金番作者被罚)对'全自动产出'的合规红线在哪——automate到什么比例不触发平台AI检测/封号, 是产品化前必须确认的政策约束。
- 人类当driver的'节奏/剧情走向'如何低摩擦介入:每章都要人确认章纲会拖垮日更效率, 能否把'人确认'降频到每卷/每arc一次, 中间章纯自动跑、只在reviewer报异常时召回人？
