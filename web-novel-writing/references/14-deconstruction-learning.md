# 14 · 拆爆款学习法 + 选题决策法

> Purpose: 给"开书前"补两块工程化的输入：**① 选题决策法**（把市场数据→"写什么能爆"的带纪律假设）+ **② 拆对标爆款学习法**（把别人的爆款逆向成"可在自己书里复用、又不抄袭"的结构模块卡 EM-card）。两者都喂进**阶段0 契约**（SKILL.md §2）：选题决策定 `config.genre/subgenre/tone/platform` + `contract.core_conflict/sao_engine` 的方向；拆对标书产出的 EM-card 填实 `contract.sao_engine`/`reader_promise`/`style.anchors`，并把"能爆的原因"从假设回填成带出处的证据。craft 借鉴 oh-story-claudecode（MIT）的 story-long-scan / story-long-analyze 两条管道，**只吸收方法论，不吸收它的抓站脚本**——本 skill 只消费结构化数据，不抓站（见 §1）。

---

## 0. 定位：这两块在流水线的哪一步、为什么是 阶段0 的前置

| craft | 回答的问题 | 输出 | 喂给 |
|---|---|---|---|
| **选题决策** | 这个题材/方向能不能爆、行不行、怎么验证 | 2-3 个带可行性档位的候选选题（"能爆的原因"先记为假设） | `config.genre/subgenre/tone/platform`（SKILL.md §2.2）+ `contract.core_conflict` 方向 |
| **拆对标书** | 爆款靠什么结构吃这波读者、我能复用哪块 | 每本对标书一组 **EM-card 可复现模块卡** + 文风样例段 | `contract.sao_engine` / `reader_promise`（§2.3）、`style.anchors`（§2.3 文风）、`emotion-debt.yaml` 的 `sao_schedule` 排布、`sao_payoff` 结构 |

**两者是一个闭环，不是两件孤立事**（沿用 oh-story scan→analyze 的回填设计）：

```
选题决策（市场侧，给"能爆的原因"=假设）
        │  标记 待拆文验证
        ▼
拆对标书（拆 ≥3-5 本同类爆款 → EM-card）
        │  EM-card = "能爆的原因"的证据
        ▼
回填：把选题里的"能爆的原因 待拆文验证" 改成带出处的支撑
        │
        ▼
锁定选定选题 + EM-card → 阶段0 契约（sao_engine / reader_promise / style.anchors）
```

> ⚠️ 铁律：**单本上榜只是个例，多本重复才算信号；拆一本对标书也只算"假设有支撑"，不算坐实**（扫榜原则1）。这条纪律贯穿 §2 选题与 §4 拆书的所有量化判断。

## 1. 数据从哪来：消费结构化数据，**不抓站**（关键边界）

oh-story 原管道自带各平台 scraper（起点 SSR / 番茄字体反爬解码 / 晋江 gb18030…）。**本 skill 一律不内置抓站脚本**——榜单与对标书原文的获取，走以下三条由用户/路由负责的来源，skill 只接"已结构化的数据"：

| 来源 | 怎么来 | 适用 |
|---|---|---|
| **① 用户直接提供** | 用户粘贴榜单截图/文字、或贴对标书章节原文、或给本地原文文件路径 | 默认首选；用户手上已有数据时 |
| **② 路由已有的浏览器工具自取** | 用 AX 已配的 `claude-in-chrome`（驱动真实 Chrome 抓榜/读章）/ `/browser`（Jina 读取渲染后内容）/ `chrome-devtools`，把页面读成结构化文本后再进本流程 | 用户没现成数据、要现抓时——**用通用浏览器工具，不写平台专用 scraper** |
| **③ 内置知识兜底** | 无榜单、无原文时，仅凭题材常识给候选 | 最弱；产物**一律标"未经榜单验证、只是候选假设"**，可行性强制封顶"中"（§2） |

**为什么切断 scraper 耦合**：平台反爬/页面结构是高频漂移的运维负担，与"约束 AI 写好长篇"的核心无关；把它绑进 skill 会让一个内容工艺 skill 背上一条会烂的抓站流水线（违背本仓库反 overengineering 原则）。采集是用户/通用浏览器工具的事，skill 的职责从"数据进来"开始。

### 数据清洗 5 条（消费用户/浏览器工具给的原始榜单时套这 5 条，仅此 5 条）

只吸收 oh-story scan 的数据清洗规则，**抓站细节一概不要**：

| # | 规则 | 做法 |
|---|---|---|
| 1 | **必填字段** | 排名/书名/作者缺任一 ⇒ 该条无效，移除后**重算条目数** |
| 2 | **字段一致性** | 同一榜单内所有条目字段集必须一致；不一致的条目标 `[字段缺失:{字段名}]` |
| 3 | **平台模板文本剔除** | 删掉"提供XXX完整版在线免费阅读""上一页"等平台模板串，保留正文 |
| 4 | **简介截断** | 先删模板文本，再把超 100 字的简介在最近的句号/问号/感叹号处截断 + `...`（⚠️100 字为经验阈值，可调） |
| 5 | **样本量底线** | 有效条目 < 15（小平台 < 10）⇒ 文件头标 `[数据稀疏] 实际 N 条`，直接触发 §2 的"不许给高"硬规则（⚠️15/10 为经验阈值，可调） |

## 2. 选题决策法：从数据到"写什么能爆"

### 2.1 选题四步（每个候选选题走完，缺一不可）

| 步 | 名 | 内容 | 纪律 |
|---|---|---|---|
| 1 | **能爆的原因（先当假设）** | 从榜上**重复出现的样本**（排除单本个例）+ 新元素，推出"这方向靠 X 结构/梗/人设吃这波读者" | 只写假设，标 `待拆文验证`；坐实靠 §4 拆对标书回填 |
| 2 | **市场验证** | 同方向榜上几本 + 趋势(↑/→/↓) + 反例（同方向有没有扑街/烂尾的） | 样本越多、趋势越稳越可信 |
| 3 | **差异化定位** | 作者优势 × 市场缺口 = "你的版本和榜上现有的不同在哪" | 无差异化 = 同质化挤不进去 |
| 4 | **可行性 + 风险 + 验证动作** | 给可行性档（§2.2）+ 最可能崩在哪 + 开写前怎么低成本验证（通常："先写黄金三章测追读，不行就换"） | 验证动作必须是低成本可执行的 |

### 2.2 可行性三档 + 硬规则（防"几条数据撑出假信心"）

| 档 | 含义 | 条件 |
|---|---|---|
| **高** | 可放心写 | 同方向样本够（≥15，小平台≥10 ⚠️经验值）+ 趋势 ↑或→ + 作者素材撑得住 + 有差异化空间 |
| **中** | 可写但先验证 | 样本够但趋势↓ / 差异化不清 / 作者素材半撑 |
| **低** | 不建议 | 同方向已饱和（扑街反例多）/ 作者素材撑不住 / 平台调性不符 |

**两条不可破的硬规则**（直接抄 oh-story 的 epistemic 纪律）：

1. **样本 < 15（小平台 < 10）⇒ 不许给"高"，强制降"中" + 写"样本不足，先扫够样本或试水再定"。** 凡背靠的榜单标了 `[数据稀疏]`（§1 清洗规则5）同样封顶"中"。
2. **内置知识模式（§1 来源③，无榜单）⇒ 所有方向一律给"中"**，理由写"基于通用认知，无榜单验证，开写前必须先补数据或小成本试水"。一律不许给"高"。

> 这两条是"样本<15 不许给高"的硬编码，**绝不能因为某方向"看起来很稳"就破例**——这正是 SKILL.md §0 铁律7"诚实标注能力边界、不夸大"在选题环节的落地。

### 2.3 选题决策产物（喂给 阶段0 契约）

落成一张轻量表（可写进 `mybook/选题决策.md` 或直接进 `contract.yaml` 注释），每个候选含：题材组合（主类型+副类型/梗）· 目标读者 · 核心卖点 · 能爆的原因(`待拆文验证`) · 差异化定位 · 可行性档+理由(写清同方向几本+趋势) · 失败风险 · 验证动作 · 篇幅/平台。2-3 个为宜，按可行性排序（高在前）。

**选定后**：把该选题的 `config.genre/subgenre/tone/platform` 写进 `contract.yaml`（触发 §05 品类模板 + §06 平台参数挂载）；"能爆的原因"先留 `待拆文验证`，等 §4 拆完回填。

## 3. 选对标书：拆谁，比怎么拆更重要

拆错样本，方法再对也学不到能复用的东西。oh-story 的对标书选择规则直接照搬：

| 别这么选 | 为什么 | 正解 |
|---|---|---|
| 只拆 1 本 | 单本规律不可靠，易把作者个人特色当通用套路 | **同平台同题材同类型至少 3-5 本**（⚠️5-10 本更稳，经验值） |
| 只拆唯一头部爆款 | 头部可能靠**不可复制**的作者品牌/历史流量/早期红利，不是结构本身 | 优先选**同平台同题材、数据稳定、非唯一头部**的可复制样本 |
| 跨平台对标 | 不同平台读者群与写法差异大（起点男频 vs 晋江女频审美天差地别，见 §06） | 对标书与你的目标平台**同平台**（沿用 §2 选定的 `config.platform`） |

流程：广泛收集同题材样本 → 定方向 → 定参考书 3-10 本 → 从中选 1-3 本**重点拆**（出 EM-card），其余略读校验模式重复性。

> **长篇连载 ADAPT（关键）**：oh-story analyze 默认**逐章拆完整本**（适合短篇/单本）。长篇连载**不必逐章拆整本对标书**——按**卷 / arc**粒度拆"①开篇黄金三章/第一卷的钩子结构 ②一个完整爽点循环单元（如打脸单元/副本单元/轮回）③章末钩子样式 ④分卷节奏（每卷起承转合 + 卷尾埋线）"。因为长篇真正要偷的是"**能在自己书里按卷复用的结构模块**"，不是整本逐章摘要。颗粒度对齐你自己的 `state-plotline.yaml`（卷/arc/章三层）。

## 4. 拆书方法论：把剧情拆成"去素材的结构"

### 4.1 六原则（拆之前先立心态）

有目的 / 有方法 / 拆出理论（不是复述剧情）/ 摒弃好恶（不带主观评价）/ 扬长避短（只学你能用的）/ 拆自己喜欢的书（有动力且品味对路）。

### 4.2 拆书四步流程

`概括成一句话` → `识别结构单元（起承转合，逐级递进扩大）` → `抽离框架 + 情绪链条` → `借鉴框架，绝不抄内容`。

### 4.3 三种"去素材"提取法（核心工具）

| 法 | 步骤 | 产出 |
|---|---|---|
| **条件框架提取法** | 把一段剧情提炼成**条件**（如：展露部分天赋 + 两方势力争夺 + 信息差制造慧眼识珠），替换人物/背景/金手指后复用，**保留情绪相似** | 一组可填空的条件 → 进 EM-card 的"戏剧单元" |
| **抽象五步法** | ①概括 → ②拆为信息团（每个=一个情绪转折点）→ ③按 **上行/下行/转折** 归类 → ④提取通用条件框架 → ⑤用自己的素材填充 | 情绪链 → 对齐 `reader_emotion_curve` + EM-card"情绪链" |
| **逐段功能标注法** | 对照原文逐段标叙事功能（铺垫/转折/高潮/过渡），追踪情绪走向、困境设计、伏笔衔接 | 可复用框架；**禁止复制剧情内容或表达原句** |

### 4.4 抽象三幕式（整本/整卷骨架归纳）

| 幕 | 核心 | 对齐 |
|---|---|---|
| 一·建立 | 暗示矛盾 → 主角+金手指 → 惊人意外 → 情绪上下拉扯 → 主线确立 | `contract.core_conflict` / `goldfinger` / 黄金三章 |
| 二·对抗 | 新世界 → 更强对手 → 中点反转 → 情绪拉扯（下行后必有上行） | 分卷节奏 / `sao_schedule` |
| 三·决战 | 线索收束 → 矛盾极限 → 关键选择（体现人物弧光）→ 决战 + 主题升华 | `mainline_anchors` 回收 / `foreshadows` closed |

> 拆出的"情绪折线图"（横轴章节、纵轴情绪正负，控制节奏=控制"造成情绪转折的信息团"出现速度）直接对照本 skill 的 `reader_emotion_curve`（章纲）与 `sao_schedule`（emotion-debt.yaml）——拆对标书是给这两个字段找"已被市场验证的排布样式"。

## 5. 可复现模块卡 EM-card：复用 + 防抄袭的硬边界

拆书的最终产物**不是剧情摘要**，而是一张张"去素材"的 EM-card——它把"读者想看什么"和"用什么结构交付"沉淀成可往自己书里重组的模块，并用 **可替换项 vs 不可照搬** 划出**防抄袭硬边界**。

### 5.1 EM-card 字段（对应 `templates/benchmark-module-card.yaml`，由主 agent 建模板）

| 字段 | 内容 | 对齐本 skill 契约 |
|---|---|---|
| `id` | EM-001… | — |
| `reader_need` | 读者想看什么（一句话抽象需求：优越感/安全感/被理解/替代性复仇/关系补偿/认知惊喜…） | `emotion-debt.yaml` 读者需求 |
| `emotion_chain` | 缺口 → 加压 → 触发 → 爆发 → 余波 | `sao_payoff.setup_chain` + `reader_emotion_curve` |
| `drama_unit` | **去素材后的结构**（如"被轻视者在公开场合用结果反证"）| 对应 §05 品类骨架 / §09 桥段标签 |
| `trigger_slots` | 可替换功能位：误判者 / 见证者 / 代价 / 证据 / 奖励 | §07 装逼三层、打脸四部曲的角色位 |
| `reproduce_steps` | 1.建缺口 2.加反应层 3.控爆发点 4.立新钩子 | 章纲 outliner 排布步骤 |
| **`swappable`（可替换项）** | 角色身份 / 场景 / 道具 / 对手类型 / 目标结果 | 这些**必须换**，换够才不雷同 |
| **`do_not_copy`（不可照搬）** | 原文专名 / 具体事件顺序 / 标志性台词 / 独特设定 | **抄袭红线**，见 §5.2 |
| `linked_sao` / `linked_hook` | 关联 `sao_schedule` 条目 / `ending_hook.type`(4型) | 挂回排布与钩子，不平行新建 |

> EM-card 模块来源标注（哪几本对标书共有此模块）尽量标，**无把握就留空，不硬凑、不删模块**（best-effort，沿用 oh-story 的桥段标签纪律）。同一模块被 ≥2 本对标书印证，才从"假设"升格为"信号"（§0 铁律）。

### 5.2 防抄袭硬边界：`do_not_copy` = 抄袭红线（reviewer 裁决细化，非新框架）

这是把 EM-card 安全用进自己书的**不可破规则**：

1. **writer 只拿 EM-card（去素材的结构），永远拿不到对标书原文段落。** prompt 编译（§03）注入的是 `drama_unit` + `swappable` 槽位，**绝不注入对标书原句**——否则 LLM 会顺手复述，等于洗稿。
2. **`do_not_copy` 项进 `contract.forbidden` 的查重红线。** reviewer（§04）增设一条硬门 `no_benchmark_copy`：本章正文若命中任一对标书的 `do_not_copy`（原文专名/标志性台词/独特设定/逐段同序）⇒ 与 `no_poison_points`/`no_forbidden_reveal_leak` 同级**直接打回**，不可被高爽点分平均掉。
3. **复现到自己书的"至少改三项"防雷同规则**：把一张 EM-card 落进自己大纲时，`swappable` 里**至少改变三项**（冲突对象 / 公开-私密场景 / 触发证据 / 余波方向）。改不够三项 = 雷同风险，回炉重组。
4. **人设优先于套路**：若 EM-card 与本书人物 `behavior_anchors` 冲突，以人设驱动重构模块，**不为套套路扭曲角色**（对齐 §02 人物卡）。

> 这条不是另起一套抄袭检测体系，而是把已有的 reviewer 硬门 + `contract.forbidden` + 人物卡**裁决细化**到"拆对标书复用"这个场景——挂在既有契约之上（SKILL.md §0 铁律4/5）。

## 6. 闭环回填 + 一页速查

### 6.1 回填：拆完对标书，把选题的"能爆的原因"坐实

拆完重点对标书、产出 EM-card 后，回到 §2 的选题决策表，把对得上题材的那个选题的"能爆的原因 `待拆文验证`"改成带出处的支撑：

> 本书拆解支撑：{对标书的 读者需求/情绪引擎 + EM-card Top 模块 + 爽点/钩子节奏样式}（出处：{对标书名 + EM-card id}）。

注意仍是"有支撑的假设"（只拆了一本/几本，不算全市场坐实）；只回填还标 `待拆文验证` 的，已填过的不动。回填后，把 EM-card 的 `reader_need` + `drama_unit` 提炼进 `contract.sao_engine`（单句锚）与 `contract.reader_promise`，拆出的文风范例段进 `style.anchors`。

### 6.2 一页速查

| 想干什么 | 看哪节 | 关键纪律/硬规则 |
|---|---|---|
| 拿到的榜单/原文怎么进流程 | §1 | 不抓站；只消费结构化数据；清洗 5 条 |
| 把市场数据变成选题 | §2 | 选题四步；样本<15 不许给高；内置知识一律给中 |
| 选哪些书来拆 | §3 | ≥3-5 本同平台同题材；不只拆唯一头部；长篇按卷/arc 拆 |
| 怎么拆成可复用结构 | §4 | 条件框架/抽象五步/逐段功能标注；拆理论不抄内容 |
| 复用又不抄袭 | §5 | EM-card 可替换 vs 不可照搬；`no_benchmark_copy` 硬门；至少改三项 |
| 拆完怎么落回契约 | §6.1 | 回填"能爆的原因"；喂 `sao_engine`/`reader_promise`/`style.anchors` |
